### 2. 프로젝트 개요
*개발 기간 : 2025.04.06 ~ 2025.05.15

NovelCraft 는 독자와 작가 모두를 위한 웹소설 양성 플랫폼입니다.

단순한 소설 감상 서비스를 넘어 신입 작가의 성장과 
독자의 독서 습관 형성을 동시에 지원하는 것이 핵심 목표입니다.

또한 성장하는 웹소설 시장에서 신입 작가들이 데뷔하고 성장할 수 있는 
체계적인 발판을 지원하기 위해 멘토/멘티 시스템을 도입, 
경력 작가와 신입 작가를 연결, 
독자에게는 다양한 독서 경험을 제공합니다.

### 10. 트러블 슈팅
<details>
<summary><b>전민우 - 멘토링 피드백 회차 동시성 이슈 — sessionNumber 중복 저장 방어</b></summary>

<br>

<aside>


## **💡배경**

</aside>

멘토링 피드백 작성 API를 구현하면서 피드백의 회차를 자동으로 부여하는 로직이 필요했다. 멘토가 멘티에게 피드백을 작성할 때마다 1회차, 2회차, 3회차... 순서로 이력이 쌓이는 구조로 설계했고, 이를 위해 `Mentorship.totalSessions + 1`을 `sessionNumber`로 사용하는 방식을 채택했다.

CodeRabbit 코드 리뷰 과정에서 이 로직이 동시 요청에 취약하다는 Critical 피드백을 받았고, 실제로 두 요청이 동시에 들어오면 같은 sessionNumber가 중복 저장되어 회차 이력 데이터가 오염될 수 있다는 것을 확인했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 문제 코드 — 동시 요청 시 같은 totalSessions 값을 동시에 읽음
int nextSession = mentorship.getTotalSessions() + 1;

MentorFeedback feedback = MentorFeedback.create(
        mentoringId,
        mentor.getId(),
        request.title(),
        nextSession,        // 요청 A → sessionNumber = 1
        request.content()   // 요청 B → sessionNumber = 1 (중복 저장!)
);
mentorFeedbackRepository.save(feedback);
mentorship.increaseSession();
```

요청 A와 요청 B가 동시에 `getTotalSessions()`를 호출하면 둘 다 0을 읽고 `nextSession = 1`을 계산한다. 결과적으로 sessionNumber가 1인 피드백이 두 개 저장되어 회차 이력이 깨진다. 더불어 `increaseSession()`도 두 번 호출되어 `totalSessions`가 실제 피드백 수와 불일치하게 된다.

---

<aside>


## 💭가설

</aside>

1. `totalSessions`를 읽고 저장하는 사이에 다른 트랜잭션이 끼어드는 것이 근본 원인이므로, 해당 row를 잠가서 접근을 직렬화하면 해결될 것이다.
2. 애플리케이션 레벨의 락만으로는 네트워크 지연, 트랜잭션 타이밍 등 예외 상황에서 여전히 중복이 발생할 수 있으므로 DB 레벨에서도 이중으로 방어해야 한다.
3. V1은 기존 동작을 유지하고 V2에서 동시성 보호를 적용해 API 경로로 버전을 분리하는 것이 팀원 영향을 최소화하는 방법이다.

---

<aside>


## ✅해결 과정

</aside>

**Step 1: 비관적 락 적용 (애플리케이션 레벨)**

멘토링 row 자체에 `PESSIMISTIC_WRITE` 락을 걸어 한 트랜잭션이 처리되는 동안 다른 트랜잭션이 같은 row를 읽지 못하도록 직렬화했다.

java

```jsx
// MentorshipRepository — 비관적 락 메서드 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Mentorship m WHERE m.id = :id")
Optional<Mentorship> findByIdWithLock(@Param("id") Long id);
```

java

```jsx
// MentoringService.createFeedbackV2() — 락 적용
// V2: findByIdWithLock — 비관적 락으로 동시 요청 직렬화
Mentorship mentorship = mentorshipRepository.findByIdWithLock(mentoringId)
        .orElseThrow(() -> new ServiceErrorException(MentoringExceptionEnum.MENTORING_NOT_FOUND));

int nextSession = mentorship.getTotalSessions() + 1;
// 락이 걸린 상태에서 읽으므로 항상 정확한 값을 읽음
```

**Step 2: DB 유니크 제약 추가 (DB 레벨 이중 방어)**

락을 피한 극단적인 동시 요청에도 중복 저장이 불가능하도록 DB 레벨에서 `(mentorship_id, session_number)` 조합에 유니크 제약을 추가했다.

java

```jsx
// MentorFeedback 엔티티 — 유니크 제약 추가
@Table(
    name = "mentorship_feedbacks",
    uniqueConstraints = @UniqueConstraint(
            columnNames = {"mentorship_id", "session_number"}
    )
)
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class MentorFeedback extends BaseEntity {

    @Column(nullable = false)
    private int sessionNumber;   // (mentorship_id, session_number) 유니크 보장

    // ...
}
```

**Step 3: 유니크 제약 충돌 시 예외 변환**

DB 레벨에서 충돌이 발생하더라도 클라이언트에게 500 대신 의미 있는 409 응답을 반환하도록 예외를 변환했다.

java

```jsx
// MentoringExceptionEnum — SESSION_CONFLICT 추가
MENTORING_SESSION_CONFLICT(HttpStatus.CONFLICT,
        "동시 요청으로 인해 피드백 등록에 실패했습니다 잠시 후 다시 시도해 주세요");
```

java

```jsx
// MentoringService.createFeedbackV2() — 예외 변환
try {
    mentorFeedbackRepository.save(feedback);
} catch (DataIntegrityViolationException e) {
    // 유니크 제약 충돌 — DB 예외를 도메인 예외로 변환해 409 반환
    throw new ServiceErrorException(MentoringExceptionEnum.MENTORING_SESSION_CONFLICT);
}
mentorship.increaseSession();
```

**Step 4: V1 / V2 API 경로 분리**

기존 팀원 코드에 영향을 주지 않도록 서비스 단일 클래스 내에서 V1/V2 메서드를 분리하고 컨트롤러에서 API 경로로 버전을 구분했다.

java

```jsx
// V1: 동시성 보호 없음 — 기존 동작 유지
@PostMapping("/api/v1/mentorings/{mentoringId}/feedbacks")
public ResponseEntity<...> createFeedbackV1(...) {
    return mentoringService.createFeedback(mentoringId, userId, request);
}

// V2: 비관적 락 + 유니크 제약 — 동시성 보호
@PostMapping("/api/v2/mentorings/{mentoringId}/feedbacks")
public ResponseEntity<...> createFeedbackV2(...) {
    return mentoringService.createFeedbackV2(mentoringId, userId, request);
}
```

## ✅배운점

</aside>

동시성 문제는 단일 스레드 환경에서는 절대 재현되지 않기 때문에 코드만 봐서는 발견하기 어렵다. CodeRabbit 리뷰가 없었다면 실제 서비스에서 데이터가 오염된 후에야 발견했을 것이다.

애플리케이션 레벨의 락만 믿는 것보다 DB 레벨에서도 유니크 제약으로 이중 방어하는 것이 훨씬 안전하다. 락이 해제된 순간의 극단적인 타이밍 이슈나 애플리케이션 장애 상황에서도 DB가 마지막 방어선이 된다.

단순히 코드가 동작하는 것에 만족하지 않고 어떤 예외 상황에서도 데이터 정합성이 깨지지 않도록 설계하는 것이 중요하다는 것을 다시 한번 깨달았다.
