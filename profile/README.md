### 2. 프로젝트 개요
*개발 기간 : 2025.04.06 ~ 2025.05.15

NovelCraft 는 독자와 작가 모두를 위한 웹소설 양성 플랫폼입니다.

단순한 소설 감상 서비스를 넘어 신입 작가의 성장과 
독자의 독서 습관 형성을 동시에 지원하는 것이 핵심 목표입니다.

또한 성장하는 웹소설 시장에서 신입 작가들이 데뷔하고 성장할 수 있는 
체계적인 발판을 지원하기 위해 멘토/멘티 시스템을 도입, 
경력 작가와 신입 작가를 연결, 
독자에게는 다양한 독서 경험을 제공합니다.

### 10. 트러블 슈팅
<details>
<summary><b>전민우 - 멘토링 피드백 회차 동시성 이슈 — sessionNumber 중복 저장 방어</b></summary>

<br>

<aside>


## **💡배경**

</aside>

멘토링 피드백 작성 API를 구현하면서 피드백의 회차를 자동으로 부여하는 로직이 필요했다. 멘토가 멘티에게 피드백을 작성할 때마다 1회차, 2회차, 3회차... 순서로 이력이 쌓이는 구조로 설계했고, 이를 위해 `Mentorship.totalSessions + 1`을 `sessionNumber`로 사용하는 방식을 채택했다.

CodeRabbit 코드 리뷰 과정에서 이 로직이 동시 요청에 취약하다는 Critical 피드백을 받았고, 실제로 두 요청이 동시에 들어오면 같은 sessionNumber가 중복 저장되어 회차 이력 데이터가 오염될 수 있다는 것을 확인했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 문제 코드 — 동시 요청 시 같은 totalSessions 값을 동시에 읽음
int nextSession = mentorship.getTotalSessions() + 1;

MentorFeedback feedback = MentorFeedback.create(
        mentoringId,
        mentor.getId(),
        request.title(),
        nextSession,        // 요청 A → sessionNumber = 1
        request.content()   // 요청 B → sessionNumber = 1 (중복 저장!)
);
mentorFeedbackRepository.save(feedback);
mentorship.increaseSession();
```

요청 A와 요청 B가 동시에 `getTotalSessions()`를 호출하면 둘 다 0을 읽고 `nextSession = 1`을 계산한다. 결과적으로 sessionNumber가 1인 피드백이 두 개 저장되어 회차 이력이 깨진다. 더불어 `increaseSession()`도 두 번 호출되어 `totalSessions`가 실제 피드백 수와 불일치하게 된다.

---

<aside>


## 💭가설

</aside>

1. `totalSessions`를 읽고 저장하는 사이에 다른 트랜잭션이 끼어드는 것이 근본 원인이므로, 해당 row를 잠가서 접근을 직렬화하면 해결될 것이다.
2. 애플리케이션 레벨의 락만으로는 네트워크 지연, 트랜잭션 타이밍 등 예외 상황에서 여전히 중복이 발생할 수 있으므로 DB 레벨에서도 이중으로 방어해야 한다.
3. V1은 기존 동작을 유지하고 V2에서 동시성 보호를 적용해 API 경로로 버전을 분리하는 것이 팀원 영향을 최소화하는 방법이다.

---

<aside>


## ✅해결 과정

</aside>

**Step 1: 비관적 락 적용 (애플리케이션 레벨)**

멘토링 row 자체에 `PESSIMISTIC_WRITE` 락을 걸어 한 트랜잭션이 처리되는 동안 다른 트랜잭션이 같은 row를 읽지 못하도록 직렬화했다.

java

```jsx
// MentorshipRepository — 비관적 락 메서드 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Mentorship m WHERE m.id = :id")
Optional<Mentorship> findByIdWithLock(@Param("id") Long id);
```

java

```jsx
// MentoringService.createFeedbackV2() — 락 적용
// V2: findByIdWithLock — 비관적 락으로 동시 요청 직렬화
Mentorship mentorship = mentorshipRepository.findByIdWithLock(mentoringId)
        .orElseThrow(() -> new ServiceErrorException(MentoringExceptionEnum.MENTORING_NOT_FOUND));

int nextSession = mentorship.getTotalSessions() + 1;
// 락이 걸린 상태에서 읽으므로 항상 정확한 값을 읽음
```

**Step 2: DB 유니크 제약 추가 (DB 레벨 이중 방어)**

락을 피한 극단적인 동시 요청에도 중복 저장이 불가능하도록 DB 레벨에서 `(mentorship_id, session_number)` 조합에 유니크 제약을 추가했다.

java

```jsx
// MentorFeedback 엔티티 — 유니크 제약 추가
@Table(
    name = "mentorship_feedbacks",
    uniqueConstraints = @UniqueConstraint(
            columnNames = {"mentorship_id", "session_number"}
    )
)
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class MentorFeedback extends BaseEntity {

    @Column(nullable = false)
    private int sessionNumber;   // (mentorship_id, session_number) 유니크 보장

    // ...
}
```

**Step 3: 유니크 제약 충돌 시 예외 변환**

DB 레벨에서 충돌이 발생하더라도 클라이언트에게 500 대신 의미 있는 409 응답을 반환하도록 예외를 변환했다.

java

```jsx
// MentoringExceptionEnum — SESSION_CONFLICT 추가
MENTORING_SESSION_CONFLICT(HttpStatus.CONFLICT,
        "동시 요청으로 인해 피드백 등록에 실패했습니다 잠시 후 다시 시도해 주세요");
```

java

```jsx
// MentoringService.createFeedbackV2() — 예외 변환
try {
    mentorFeedbackRepository.save(feedback);
} catch (DataIntegrityViolationException e) {
    // 유니크 제약 충돌 — DB 예외를 도메인 예외로 변환해 409 반환
    throw new ServiceErrorException(MentoringExceptionEnum.MENTORING_SESSION_CONFLICT);
}
mentorship.increaseSession();
```

**Step 4: V1 / V2 API 경로 분리**

기존 팀원 코드에 영향을 주지 않도록 서비스 단일 클래스 내에서 V1/V2 메서드를 분리하고 컨트롤러에서 API 경로로 버전을 구분했다.

java

```jsx
// V1: 동시성 보호 없음 — 기존 동작 유지
```

## **💡배경**

</aside>

웹소설 플랫폼 특성상 메인 페이지의 '신작 소설 목록'은 사용자가 가장 먼저, 그리고 가장 많이 접근하여 트래픽이 집중되는 핵심 기능이라고 생각한다.

초기 구현 시에는 단순 JPA 메서드와 Redis 캐싱만을 활용하여 "최근 30일 이내 생성된 소설"을 조회하도록 기능을 구현해두었던게 발단이었다.

<aside>


## 🚨문제 상황

</aside>

테스트 및 튜터링(멘토링) 과정에서 

> "현재는 데이터가 적어 빠르지만, 서비스가 성장하여 소설 데이터가 수십~수백만 건으로 쌓이게 되면 병목이 발생할 것"
> 

이라는 피드백을 받아 기존 코드를 분석해보니 `is_deleted=false, status='ONGOING', created_at >= 30일` 등의 조회 조건은 걸려 있었으나, **적절한 인덱스를 타지 못해 데이터베이스 전체를 뒤지는 Full Table Scan**이 발생할 위험이 있었다.

<aside>


## 💭가설

</aside>

1. 데이터베이스 계층의 인덱스 부재가 대용량 데이터 조회 시 가장 큰 성능 저하 요인이 될 것이다.
2. 메인 페이지의 신작 목록은 실시간성(초 단위)이 극도로 요구되지 않아 DB 조회 횟수를 최소화하고 캐싱을 적극적으로 활용하면 성능을 안정적으로 유지할 수 있을 것이다.
3. 성능 최적화는 한 번에 오버엔지니어링(예: 샤딩 도입)을 하기보다 `"DB 최적화(인덱싱) → 애플리케이션 최적화(QueryDSL) → 인프라 최적화(Redis 캐싱)"` 순으로 병목 지점을 해결하는 것이 올바른 접근일 것이다.

## ✅해결 과정

</aside>

#### **✅ Step 1: DB 최적화 (복합 인덱스 적용)**

데이터 중복도(카디널리티)와 조회 조건을 고려하여, 조건에 일치하는 데이터만 스캔하도록 Novel 엔티티에 복합 인덱스를 추가했다.

```java
@Table(name = "novels",
        indexes = {
                @Index(name = "idx_novel_genre", columnList = "genre"),
                @Index(name = "idx_novel_status", columnList = "status"),

                // 신작 소설 조회용 복합 인덱스
                @Index(name = "idx_novel_new_list", columnList = "is_deleted, status, genre, created_at")
        })
public class Novel extends BaseEntity {
```

#### **✅ Step 2: 캐싱 전략 고도화 (Look-Aside 패턴)**

- 실시간성에 목매지 않아도 되는 신작 리스트의 특성을 살려, 조회 시 Redis를 먼저 확인(Cache Hit)하고 데이터가 없을 때만 DB를 조회(Cache Miss)하는 Look-Aside 패턴을 구현
- 장르, 상태, 제한 개수(limit)를 조합한 Cache Key(`NOVEL_LIST_CACHE_KEY + "type:new::" + "genre:" + ...`)를 설계하고 적절한 TTL(30분)을 부여하여 무의미한 DB 접근을 차단했다.

```java
// 신작용 소설 목록 조회 (QueryDSL+Redis캐싱+인덱싱)
    // 한 달 신작 리스트 (limit 50 작품)
    @Transactional(readOnly = true)
    public List<NovelListResponse> getNewNovelList(String genre, NovelStatus status, int limit) {

        String cacheKey = NOVEL_LIST_CACHE_KEY
                + "type:new::"
                + "genre:" + genre + "::"
                + "status:" + status + "::"
                + "limit:" + limit;

        // Redis 캐시 확인
        Object cached = null;
        try {
            cached = redisTemplate.opsForValue().get(cacheKey);
        } catch (RuntimeException e) {
            log.warn("소설 리스트 캐시를 읽어내는데 실패했습니다. key={}", cacheKey, e);
        }

        if (cached != null) {
            log.debug("===== [신작 캐시 HIT] key={} =====", cacheKey);
            return (List<NovelListResponse>) cached; // List 타입 캐스팅
        }

        log.debug("===== [신작 캐시 MISS] key={} DB 조회 =====", cacheKey);

        // 3. DB 조회 (캐시에 없을 때만)
        List<NovelListResponse> response = novelRepository.findNewNovelList(genre, status, limit);

        // 4. Redis 캐시에 결과 저장
        try {
            redisTemplate.opsForValue().set(cacheKey, response, NOVEL_LIST_CACHE_TTL);
            log.debug("===== [신작 캐시 저장] key={} TTL=30분 =====", cacheKey);
        } catch (RuntimeException e) {
            log.warn("New novel list cache write failed. key={}", cacheKey, e);
        }

        return response;
      }
```

#### **✅ Step 3: QueryDSL을 통한 쿼리 최적화**

QueryDSL의 `Projections.constructor`를 활용하여 엔티티 전체가 아닌 DTO 형태로 꼭 필요한 필드만 Select 하도록 구성하고 Fetch Join을 통해 N+1 문제를 방지했다.

```java
// 신작용 소설 목록 조회 (한 달 신작 리스트)
    @Override
    public List<NovelListResponse> findNewNovelList(String genre, NovelStatus status, int limit) {
        QNovel novel = QNovel.novel;
        QUser user = QUser.user;

        // 한 달 전 날짜 계산
        LocalDateTime oneMonthAgo = LocalDateTime.now().minusMonths(1);

        return queryFactory
                .select(Projections.constructor(NovelListResponse.class,
                        novel.id,
                        novel.title,
                        novel.genre,
                        novel.tags,
                        novel.status,
                        novel.coverImageUrl,
                        novel.viewCount,
                        novel.bookmarkCount,
                        user.nickname
                        ))
                .from(novel)
                .join(user).on(novel.authorId.eq(user.id))
                .where(
                        novel.isDeleted.eq(false)
                        , novel.status.ne(NovelStatus.PENDING)
                        , genreEq(genre)
                        , statusEq(status)
                        , novel.createdAt.goe(oneMonthAgo) // 한 달 이내 신작 소설들
                )
                .orderBy(novel.createdAt.desc())
                .limit(limit)
                .fetch();

    }
```

## ✅배운점

</aside>

Full Table Scan으로 인해 발생할 수 있었던 메인 페이지 병목 현상을 복합 인덱스와 QueryDSL을 통해 안전하게 방어하면서

그리고 무작정 파티셔닝이나 샤딩 같은 고도의 기술을 맹목적으로 도입하기보다는 **처음부터 천천히 현재 트래픽 규모에 맞는 기본기(Index + Cache)**부터 단계적으로 성능을 끌어올리는 인프라 설계를 머릿속에 새길 수 있었다.

    - **Redis 원자성 보장 성공** 
    단순 인증 로직에 무거운 분산 락을 거는 대신 싱글 스레드로 동작하는 Redis의 특징을 활용한 Lua Script로 동시성 취약점을 완벽히 해결하고 불필요한 SMS 과금 방어

분산 락 대신 오버헤드가 적은 **Redis Lua Script**를 도입하여 GET과 DEL 명령어를 묶어 원자적(Atomic)으로 처리했다.

그 결과 500개의 동시 검증 요청 중 단 1개의 요청만 정상 처리(200 OK)되고, 나머지 499개의 요청은 인증번호가 이미 삭제되어 실패(4xx 에러) 처리됨을 확인했다.

- **시나리오 B - 동시 회원가입 플로우**
    - **성능 테스트 고도화 경험** 
    단순 API 테스트에 그치지 않고 k6를 통한 부하 테스트를 직접 스크립팅해 정합성 보장뿐만 아니라 시스템 한계(스레드 풀, 병목 지점)를 시각적으로(Grafana 대시보드) 모니터링하고 파악하는 실무적인 성능 검증

다수의 유저가 동시에 들어왔을 때, 스프링 부트(Tomcat)의 기본 스레드 풀(200개) 고갈 및 Redis 커넥션 풀이 꽉 차는 현상으로 인해 일부 유저가 타임아웃 및 500 에러를 반환받으며 튕겨 나가는 병목 현상을 발견할 수 있었고

부하를 100% 소화하지 못하는 상황을 통해 서비스 인프라의 임계점을 파악할 수 있었다.
