## 2. 프로젝트 개요
*개발 기간 : 2025.04.06 ~ 2025.05.15

NovelCraft 는 독자와 작가 모두를 위한 웹소설 양성 플랫폼입니다.

단순한 소설 감상 서비스를 넘어 신입 작가의 성장과 
독자의 독서 습관 형성을 동시에 지원하는 것이 핵심 목표입니다.

또한 성장하는 웹소설 시장에서 신입 작가들이 데뷔하고 성장할 수 있는 
체계적인 발판을 지원하기 위해 멘토/멘티 시스템을 도입, 
경력 작가와 신입 작가를 연결, 
독자에게는 다양한 독서 경험을 제공합니다.

## 8. 기술적 의사 결정

<details>
<summary><b>🟦User - 단계별 회원가입 Insert 로직 재설계</b></summary>

<aside>


## **💡배경**

</aside>

테스트 과정에서 사용자가 1단계만 진행하고 이탈한 유저나 탈퇴 후 복구 테스트할 때 자꾸 찌꺼기 데이터가 남아 버그가 생겼나? 생각이 들었다.

로직을 역추적해보니 초기 설계 의도와 달리 1단계 회원가입에서 DB에 강제 저장이 되고 있었다.

<aside>


## **🚨문제 정의**

</aside>

#### **1. Row Update 방식의 한계**

1단계 회원가입 완료 시 DB에 더미 데이터를 먼저 Insert 하고 2단계 완료 시 해당 데이터를 update 하는 방식

#### **2. 더미 데이터 누적**

특히 1단계 회원가입을 진행하고 중도 이탈할 경우 DB에 Temp 상태로 DB에 불필요한 쓰레기 데이터가 지속적으로 쌓이는 문제 발견	

#### **3. 잠재적 시스템 버그 유발**

남아있던 TEMP 더미 데이터들로 인해 로그인 및 기타 비즈니스 로직에서 인가처리 등 예기치 않은 오류가 발생할 위험성이 있었음

```java
// 1단계: 아직 가입 완료 전인데 영구 저장소(DB)에 강제 저장 발생
User user = User.register(..., UserRole.TEMP);
userRepository.save(user); 
userRepository.flush(); // 문제 원인: 이탈 시 쓰레기 데이터 생성!

// 2단계: 기존 데이터를 찾아 권한만 변경 (Row Update)
user.changeRole(UserRole.READER);
```

<aside>


## **✅해결 방안**

</aside>

1단계 입력 정보는 아직 가입이 확정되지 않은 휘발성이 강한 데이터이므로 영구 저장소인 DB에 보관하는 것은 부적절하다고 판단.

만료 시간 설정이 가능한 메모리 기반의 Redis를 도입하면 불필요한 쓰레기 데이터 누적을 방지함과 동시에 별도의 스케쥴러 운영을 하지 않아 리소스를 최적화 할 수 있음과 I/O 성능까지 향상 시킬 수 있을 것 같았다.

- **Redis 도입 및 TTL 설정** :
    - 임시 데이터 저장소로 Redis를 선택
    - Key-Value 형태로 데이터를 보관하고, 지정된 시간 후 자동 삭제되도록 아키텍쳐 수정

```java
// 1단계: DB 접근 없음. Redis에 DTO를 통째로 10분(TTL) 임시 저장
String redisKey = "TEMP_SIGNUP:" + request.email();
redisUtil.set(redisKey, tempRequest, 10); 

// 2단계: 추가 정보 입력 완료 시점에 Redis에서 꺼내와 최초 1회 DB 저장
SignupRequest tempRequest = (SignupRequest) redisUtil.get(redisKey);
User savedUser = userRepository.save(user);
```

<aside>


## **♻️구현 흐름**

</aside>

#### **1. 공통 회원가입 1단계 - Redis 임시 저장**

1단계 필수 가입 정보 제출 시, DB가 아닌 Redis에 공통 가입 정보를 15분 동안만 정보를 **임시 저**

#### **2. 임시 토큰(TempToken) 발급**

클라이언트에게는 실제 데이터 대신 식별 가능한 임시 토큰만 발급해 보안성을 높임

#### **3. 작가/독자 회원가입 2단계 - 데이터 병합**

2단계 추가 가입 정보 제출 시, 클라이언트가 전달한 임시 토큰(TempToken)으로 Redis에서 1단계 정보를 불러옴

#### **4. 최종 DB에 Insert**

Redis에서 가져온 1단계 정보와 새롭게 입력받은 2단계 정보를 병합, DB에 딱 한 번의 최종 Insert(저장) 수행

#### **5. 임시 토큰(TempToken) 데이터 소멸**

DB 저장이 성공적으로 완료되면 Redis에 남은 임시 데이터는 즉시 삭제 처리

## **📊해결 및 결과**

</aside>

#### **1. 데이터 무결성 확보**

가입 중도 이탈자의 데이터가 DB에 전혀 남지 않아 데이터베이스 오염 차단

#### **2. 서버 리소스 최적화**

가비지 데이터(불필요 데이터) 삭제를 위한 별도의 스케쥴러 없이 Redis TTL로 자동화하여 서버 유지보수 비용 절감

#### **3. 시스템 안정성 강화**

Temp 더미 데이터로 인해 발생할 수 있었던 잠재적 로그인 버그 요소 제거

## **[배경]**

Redis 기반 JWT 블랙리스트 운영 중, Redis 다운 시 JwtFilter 전체가 500 에러를 반환해 모든 사용자가 서비스 이용 불가능한 상태를 발견했다.

## **[문제 정의]**

Redis 장애 상황에서 로그아웃된 탈취 토큰이 블랙리스트 검증 없이 통과될 수 있는 보안 공백 발생

가용성(서비스 유지) vs 보안(탈취 토큰 차단) 사이의 트레이드오프 존재

## **[의사 결정]**

"모든 API를 동일하게 처리할 필요는 없다" 는 관점에서 API 성격별 이중 정책 적용

- `Fail-Open` (가용성 우선): 소설 읽기, 랭킹 등 조회성 API → 장애 시 통과 허용
- `Fail-Closed` (보안 우선): 결제, 회원정보 수정 등 민감 API → 장애 시 503 반환 차단

## **[구현 내용]**

`JwtFilter` 내 catch 블록에서 `isCriticalApi()` 메서드로 URL 분기 처리

→ 중요 API 리스트 관리로 정책 변경 시 코드 수정 최소화

```java
public class JwtFilter extends OncePerRequestFilter {

// 기존 로직 ...

   @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain)

// 기존 인증 로직 ... 

        String accessToken = jwtUtil.substringToken(authorizationHeader);

        if (jwtUtil.validateToken(accessToken)) {

            // Redis 블랙리스트 검사
            try {
                // Redis 정상 동작
                if (redisUtil.isBlackList(accessToken)) {
                    log.warn("블랙리스트에 등록된 토큰입니다.");
                    response.setCharacterEncoding("UTF-8");
                    sendErrorResponse(response, HttpServletResponse.SC_UNAUTHORIZED, "토큰이 유효하지 않습니다.");
                    return;
                }
            } catch (org.springframework.data.redis.RedisConnectionFailureException e) {
                // Redis 장애 상황 발생 (선택적 차단 로직 동작)
                log.error("[Redis 장애] 블랙리스트 확인 불가, URL: {}", requestURL);

                // 중요 보안 API (결제, 수정, 삭제 등) : Fail-Closed 무조건 차단
                if(isCriticalApi(requestURL)) {
                    log.error("[Redis 장애] 보안을 위해 중요 API 접근을 차단합니다. URL: {}", requestURL);
                    sendErrorResponse(response, HttpServletResponse.SC_UNAUTHORIZED, "서버 불안정으로 해당 기능을 사용할 수 없습니다.");
                    return;
                }

                // 일반 조회 API (소설 읽기, 랭킹 보기 등) : Fail-Open 가용성을 위해 허용
                log.warn("[Redis 장애] 가용성을 위해 일반 API 접근을 허용합니다. URL: {}", requestURL);
            } catch (Exception e) {
                log.error("Redis 검증 중 알 수 없는 에러 발생", e);
                sendErrorResponse(response, HttpServletResponse.SC_UNAUTHORIZED, "서버 오류가 발생했습니다.");
                return;
            }
                if (!setAuthentication(response, accessToken, requestURL)) {
                    return;
                }
                filterChain.doFilter(request, response);
                return;

        }

// 기존 토큰 확인 로직 유지...

// 제일 하단에 차단할 중요 API 리스트 판별 메서드
    private boolean isCriticalApi(String url) {
        return url.startsWith("/api/payments/prepare") ||       // 결제 준비
                url.startsWith("/api/payments/confirm") ||      // 결제 확인
                url.startsWith("/api/payments/refunds") ||      // 환불 하기
                url.startsWith("/api/revenue/me/exchanges") ||  // 환전 신청
                url.startsWith("/api/auth/signup/reader") ||    // 비밀번호 변경
                url.startsWith("/api/users/me/reader") ||       // 회원정보수정 (작가)
                url.startsWith("/api/auth/users/me/author") ||  // 회원정보수정 (독자)
                url.startsWith("/api/auth/users/me") ||         // 회원정보수정 (공통)
                url.startsWith("/api/admin/**");                // 관리자
    }
}
```

## 📚 배경

NovelCraft 결제 시스템은 포트원(PortOne) V2 외부 API와 연동된다. 클라이언트의 `/confirm` 요청과 포트원이 서버로 직접 전송하는 웹훅(Webhook)은 **동일한 결제 건에 대해 거의 동시에 도달**할 수 있다. 또한 환불 버튼을

빠르게 여러 번 누르는 등 **중복 요청**도 현실적으로 발생한다.

외부 API 호출(포트원 SDK)은 응답까지 수 초가 소요되는 I/O 블로킹 작업이다. 이 긴 대기 시간 동안 동일한 결제 건에 대한 두 번째 요청이 들어오면 데이터 정합성이 깨질 수 있다는 문제를 인식했다.

---

## 🚨 문제 정의

### 1. /confirm ↔ 웹훅 경쟁 조건 (Race Condition)

결제 완료 시 클라이언트의 `/confirm`과 포트원 웹훅이 거의 동시에 서버에 도달하는 경우가 발생한다. 둘 다 포인트 충전 + COMPLETED 상태 전환 로직을 실행하면 포인트가 **이중 충전**된다.

클라이언트   →  POST /confirm  →  포트원 SDK 조회(~3초)  →  포인트 +10,000P  →  COMPLETED

포트원 웹훅  →  handleWebhook  →  포트원 SDK 조회(~3초)  →  포인트 +10,000P  →  COMPLETED
                               ↑ 동시 진입 가능                                                    ↑ 이중 충전 발생

### 2. 환불 중복 요청 → 포인트 이중 차감

환불 흐름은 **포인트 선차감 → 포트원 환불 API 호출** 순서로 설계되어 있다. 동일한 결제에 환불 요청이 두 번 들어오면 포인트가 **두 번 차감**된다.

요청 1  →  포인트 -10,000P  →  포트원 SDK(~10초 대기 중)

요청 2  →  포인트 -10,000P  →  포트원 SDK 호출 → 이미 취소됨 오류
                         ↑ 선차감이 두 번 발생

### 3. 트랜잭션 안에서 해결이 불가능한 이유

비관적 락과 낙관적 락 모두 이 구조에 적합하지 않다.

- **비관적 락** : `SELECT ... FOR UPDATE`로 행을 잠그면 포트원 외부 API 호출(수 초) 동안 DB 커넥션과 락이 계속 점유된다. 트래픽이 몰릴 경우 커넥션 풀이 고갈되어 **전혀 무관한 다른 요청까지 지연**된다.
- **낙관적 락** : `@Version`으로 커밋 시점에 충돌을 감지하지만, 충돌이 감지되는 시점은 **포트원 API 호출이 이미 완료된 이후**다. 포인트 이중 충전이나 이중 API 호출 자체를 막지 못하고, 실패 후 보상 트랜잭션을 직접
구현해야 한다.

---

## ✅ 해결 방안

외부 API 호출은 트랜잭션 밖에서 실행되어야 하므로, **DB 트랜잭션과 완전히 분리된 레벨에서 상호 배제**가 필요하다. Redis 분산 락이 이 조건에 가장 적합하다.

- 트랜잭션 시작 전에 락을 먼저 획득 → 외부 API 호출 중복 자체를 원천 차단
- DB 커넥션을 점유하지 않음 → 커넥션 풀 부담 없음
- TTL 기반이므로 서버 장애 시 자동 해제 → 데드락 없음
- 분산 서버 환경에서도 동일하게 동작

### Redisson을 선택한 이유 (vs Lua Script + Lettuce)

SMS 인증처럼 단순한 원자적 연산(`INCR + EXPIRE`, `GET + DEL`)에는 Lua 스크립트가 충분하다. 그러나 결제 락은 **포트원 API 응답 시간이 가변적(최대 10~20초)** 이라는 특성이 있어 Lua + Lettuce로는 아래 문제가 발생한다.

| 항목 | Lua + Lettuce | Redisson |
| --- | --- | --- |
| TTL 만료 위험 | 고정 TTL 설정 필요. 외부 API가 늦어지면 락이 만료되어 중복 실행 발생 | Watchdog이 TTL 자동 갱신. 작업이 끝날 때까지 락 유지 |
| 락 소유권 검증 | value를 토큰으로 관리하는 Lua 스크립트를 직접 작성해야 함 | `isHeldByCurrentThread()`로 현재 스레드만 해제 가능. 재진입 락 기본 지원 |
| JVM 종료 처리 | 별도 처리 없으면 고아 락 발생 가능 | JVM 종료 시 Watchdog도 중단 → TTL 만료 후 자동 해제 |
| 구현 복잡도 | Lua 스크립트, 토큰 생성, TTL 갱신 로직을 직접 구현 | `tryLock()` / `unlock()` 두 줄로 완성 |

```java
// 락 획득 - 대기 없이 즉시 시도, 성공 시 Watchdog이 TTL 자동 갱신
public boolean acquireLock(String key) {
    boolean acquired = redissonClient.getLock(key).tryLock();
    return acquired;
}

// 락 해제 - 현재 스레드가 보유한 락만 해제 (소유권 검증)
public void releaseLock(String key) {
    RLock lock = redissonClient.getLock(key);
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

## ♻️ 구현 흐름

락 키는 결제 건 단위로 설계하여, 서로 다른 결제 건은 병렬 처리되고 동일한 결제 건만 직렬화된다.

payment:confirm:lock:{paymentId}      →  /confirm ↔ 웹훅 보정 상호 배제
payment:cancel:lock:{paymentId}       →  환불 중복 요청 차단 + 웹훅 환불 보정

결제 확인 흐름 (confirmPayment)

1. Redis Lock 획득 (payment:confirm:lock:{paymentId})
└─ 실패 시 → 이미 처리 중 예외 반환
2. [짧은 TX] 중복 결제 체크 + PENDING 조회
3. [트랜잭션 밖] 포트원 SDK 결제 조회 (DB 커넥션 미점유)
4. 금액 위변조 검증
5. [짧은 TX] COMPLETED 전환 + 포인트 충전 + 이력 저장
6. finally → Redis Lock 해제

환불 흐름 (cancelPayment)

1. Redis Lock 획득 (payment:cancel:lock:{paymentId})
└─ 실패 시 → 이미 처리 중 예외 반환
2. [읽기 TX] 결제 조회 + 환불 가능 상태 검증
3. [TX] 포인트 선차감 (잔액 부족 시 여기서 실패 → 포트원 미호출)
4. [트랜잭션 밖] 포트원 SDK 환불 요청
└─ 실패 시 → 선차감 포인트 복구 (보상 트랜잭션)
5. [짧은 TX] REFUNDED 상태 전환
6. finally → Redis Lock 해제

웹훅 흐름 (handleWebhook)

1. [TX] 멱등성 키(transactionId / cancellationId)로 중복 웹훅 필터링
2. [트랜잭션 밖] 포트원 SDK 조회 (위조 방지)
3. /confirm과 동일한 락 키 획득 시도
└─ 실패 시 → /confirm이 처리 중 → 그냥 리턴 (포트원이 재시도 전송)
4. PENDING 상태인 경우에만 보정 처리 (최종 상태면 건너뜀)
5. finally → Redis Lock 해제

## 📊 해결 및 결과

1. /confirm ↔ 웹훅 완전 상호 배제

두 요청이 동일한 락 키를 공유하므로, 먼저 도착한 쪽이 락을 획득하여 처리하고 나머지는 즉시 리턴된다. 포인트 이중 충전이 구조적으로 불가능해졌다.

1. 환불 중복 요청 차단

포인트 선차감 전에 락을 획득하므로, 두 번째 환불 요청은 락 획득 단계에서 즉시 차단된다. 포트원 API 자체가 호출되지 않아 보상 트랜잭션이 필요 없다.

1. DB 커넥션 효율 유지

락이 DB 외부(Redis)에 있으므로 포트원 외부 API 대기 시간(최대 10~20초) 동안 DB 커넥션이 점유되지 않는다. 결제 처리 중에도 다른 API 요청이 정상적으로 DB 커넥션을 사용할 수 있다.

1. Watchdog으로 락 안전성 확보

포트원 API가 예상보다 늦게 응답하더라도 Watchdog이 TTL을 자동 갱신하여, 처리 중에 락이 만료되는 상황이 발생하지 않는다.

## **📚배경**

</aside>

웹소설 플랫폼 특성상 인기 소설의 특정 회차는 조회 요청이 몰릴 수 있다. 이것을 대비해 회차 본문 조회에 Redis 캐싱(V2)을 도입하면 DB 부하를 줄일 수 있을 것이라 판단했다.

하지만 K6 부하테스트(VU 1000) 결과가 예상과 정반대로 나왔다. 캐시 히트율이 100%인데도 V1(JPA 직접 조회)이 V2(Redis 캐싱)보다 27% 더 빠른 결과가 나온 것이다.

<aside>


## 🚨문제 정의

</aside>

### 1. 캐싱 도입 후 오히려 성능 저하

| 지표 | V1 (JPA) | V2 (캐싱) |
| --- | --- | --- |
| 평균 응답시간 | 184ms | 233ms |
| TPS | 2304/s | 1817/s |

캐시 히트율 100%인데 V1이 더 빠른 역설적인 상황이 발생했다.

### 2. 원인 분석

**V1 흐름:**

```
DB SELECT (PK 인덱스, 1ms) → 응답
```

**V2 흐름:**

```
Redis GET → JSON 역직렬화 → 핫키 카운터 → 어뷰징 체크 → 조회수 증가 → 응답
```

회차 본문 조회는 `findById` PK 단건 조회라 DB 비용 자체가 1ms 이내로 매우 낮다. 반면 V2는 캐시 히트해도 Redis 네트워크 통신 + JSON 역직렬화 + 핫키 카운터 + 어뷰징 체크 + 조회수 증가 등 부가 로직 오버헤드가 누적되어 오히려 더 느려진 것이다.

즉 **캐싱이 느린 게 아니라, 이미 충분히 빠른 쿼리에 불필요한 오버헤드를 더한 것**이 문제였다.

<aside>


## ✅해결 방안

</aside>

튜터님 가이드대로 **순차적 성능 개선 원칙**을 따르기로 했다.

```
1. 성능 저하 원인 파악
2. 인덱스 점검
3. 쿼리 점검
4. 그래도 부족하면 캐싱 도입
```

회차 본문 조회는 PK 단건 조회라 V1 자체가 이미 충분히 최적화된 상태였다. V2에서 속도가 느린 건 캐싱 자체의 문제가 아니라 조회수 어뷰징 방지 + 핫키 카운터 같은 부가 로직이 같이 붙어 있어서 단순 성능 비교가 불가능한 구조였다.

<aside>


## ♻️구현 흐름

</aside>

### 캐싱이 의미 있는 케이스 vs 아닌 케이스

```
[캐싱이 효과적인 경우]
복잡한 JOIN 쿼리 / 집계 쿼리 / 여러 테이블 조회
→ DB 비용이 크기 때문에 캐싱으로 절감 효과 큼

[캐싱이 역효과인 경우]
PK 단건 조회 (1ms 이내)
→ DB 비용 < Redis 오버헤드
→ 오히려 느려짐
```

### 올바른 성능 개선 순서

```
인덱스 점검 → 쿼리 점검 → 캐싱 도입
```

각 단계에서 충분한 성능이 나오면 다음 단계로 넘어갈 필요 없다.

<aside>


## 📊해결 및 결과

</aside>

| 항목 | 내용 |
| --- | --- |
| 캐싱 도입 여부 | V2 캐싱 방식 제거 |
| 이유 | PK 단건 조회는 이미 충분히 빠름 |
| 대신 적용한 것 | Projections으로 필요 컬럼만 조회 |
| 캐싱 적용 대상 | 복잡한 집계 쿼리, 트래픽 몰리는 목록 조회 등으로 한정 |

<aside>


## 💡배운 점

</aside>

캐싱은 만능이 아니다. DB 비용이 비쌀 때만 의미 있는 최적화다. PK 단건 조회처럼 이미 충분히 빠른 쿼리에 캐싱을 도입하면 Redis 통신 오버헤드가 더 커서 오히려 느려질 수 있다는 걸 직접 수치로 확인했다.

성능 개선을 할 때 "캐싱하면 무조건 빠르다"는 선입견을 버려야 한다는 걸 느꼈다. 인덱스 → 쿼리 → 캐싱 순서대로 단계적으로 접근하고, 각 단계에서 충분한 성능이 나오면 굳이 다음 단계로 갈 필요가 없다. 트래픽이 몰리는 곳, DB 부하가 진짜 큰 곳에 선별적으로 캐싱을 적용하는 게 맞는 방향이라는 걸 알았다.

<summary><b>🟦ReviewAI - AI 리뷰 동기 → Kafka 비동기 전환 및 안정성 강화</b></summary>

<aside>


## **📚배경**

</aside>

AI 리뷰 기능 v1은 OpenAI를 동기로 호출해서 응답까지 평균 6초를 대기했다. 작가가 버튼을 누르면 6초 동안 화면이 멈추는 UX 문제가 있었고, 동시에 여러 작가가 요청하면 톰캣 스레드가 OpenAI 응답 대기로 점유되어 스레드풀 고갈 위험도 있었다.

이를 해결하기 위해 Kafka 기반 비동기(v2)로 전환했지만, 전환 이후 새로운 안정성 문제들이 발견됐다. 성능은 잡았지만 동시성과 외부 API 장애 대응이 미흡한 상태였다.

<aside>


## 🚨문제 정의

</aside>

### 1. 동기 처리로 인한 성능 한계

```json
v1 흐름: 요청 → 톰캣 스레드 점유 → OpenAI 호출(6초) → 응답
```

- API 응답 시간이 OpenAI 응답 시간에 완전히 종속
- 톰캣 기본 200 스레드 기준, 동시 200명만 요청해도 스레드풀 고갈
- 사용자가 버튼 누르고 6초간 화면 멈춤 → UX 저하
- AI 리뷰는 작가 본인만 보는 비실시간 작업인데 굳이 동기로 처리할 이유가 없음

### 2. 포인트 중복 차감 위험

작가가 버튼을 빠르게 2번 클릭하면:

```json
요청 A, 요청 B 동시 도착
       ↓
둘 다 포인트 잔액 체크 통과 (200P 있음)
       ↓
둘 다 처리 → 포인트 400P 차감 ❌
```

비동기 환경에서는 DB 비관적 락으로 막을 수 없어서 별도 처리가 필요했다.

### 3. Kafka 메시지 중복 소비

Kafka는 at-least-once 보장 특성상 같은 메시지가 2번 소비될 수 있다. 처리 중 Consumer가 죽으면 재시작 후 같은 메시지를 다시 처리해서 포인트가 중복 차감될 수 있었다.

### 4. OpenAI 일시 장애 대응 없음

```json
OpenAI 호출 실패 → 즉시 FAILED 처리
→ 작가는 200P 날리고 다시 요청해야 함
→ 일시 장애인데 재시도 한 번도 없이 실패는 너무 가혹함
```

<aside>


## ✅해결 방안

</aside>

### 1. Kafka 비동기 전환 (성능 문제 해결)

```json
v2 흐름: 요청 → 즉시 jobId 반환(50ms)
         백그라운드: Consumer가 OpenAI 호출 → Redis에 결과 저장(TTL 1시간)
         클라이언트: jobId로 폴링하여 결과 조회
```

### 2. Redisson 분산락 (포인트 중복 차감 방지)

```java
RLock lock = redissonClient.getLock("lock:ai-review:" + episodeId);

try {
    boolean acquired = lock.tryLock(0, 3, TimeUnit.SECONDS);
    if (!acquired) {
        throw new ServiceErrorException(AI_REVIEW_LOCK_ACQUIRE_FAILED);
    }
    // 실제 처리
} finally {
    if (lock.isHeldByCurrentThread()) lock.unlock();
}
```

- `waitTime = 0` : 락 못 얻으면 즉시 거절 (AI 리뷰는 처리 시간이 길어서 대기시켜봤자 의미 없음)
- `leaseTime = 3초` : 자동 해제로 데드락 방지

### 3. Kafka 멱등성 처리 (중복 소비 방지)

```java
// processReview() 시작 부분에 상태 체크
AiReviewJob currentJob = findJob(jobId);
if (currentJob == null) return;  // Job 없으면 스킵
if (currentJob.status() != PROCESSING) return;  // 이미 처리됐으면 스킵
```

PROCESSING 상태일 때만 처리해서 같은 메시지가 2번 와도 1번만 처리되도록 멱등성 확보

### 4. Spring Retry 폴백 처리 (OpenAI 장애 대응)

```java
@Retryable(
    retryFor = RuntimeException.class,
    maxAttempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2.0)
)
public AiReviewResponse generate(...) { ... }

@Recover
public AiReviewResponse recover(RuntimeException e, ...) {
    throw new IllegalStateException("AI 리뷰 생성에 실패했습니다.");
}
```

- 1초 → 2초 → 4초 간격으로 최대 3회 자동 재시도
- 3회 다 실패하면 Job FAILED 처리 (포인트 차감 X)

<aside>


## **💡배경**

</aside>

멘토링 피드백 작성 API를 구현하던 중, 피드백의 회차(sessionNumber)를 `totalSessions + 1`로 계산하는 로직에서 문제가 생길 수 있다는 것을 인식했다. 실제로 CodeRabbit 리뷰에서도 동일한 문제를 Critical로 지적했다. 두 요청이 동시에 들어오면 같은 sessionNumber가 중복 저장되어 회차 정렬이 깨지고, Mentorship.totalSessions와도 불일치가 발생할 수 있는 구조였다.

---

<aside>


## **🚨문제 정의**

</aside>

**1. sessionNumber 중복 저장 가능성**

현재 로직은 totalSessions를 읽고 +1한 값을 sessionNumber로 사용한다. 두 요청이 동시에 같은 값을 읽으면 동일한 sessionNumber가 두 개의 피드백에 저장된다.

java

```jsx
// 문제 코드 — 동시 요청 시 같은 nextSession 값 읽기 가능
int nextSession = mentorship.getTotalSessions() + 1;  // 요청 A, B 둘 다 1을 읽음

MentorFeedback feedback = MentorFeedback.create(
        mentoringId, mentor.getId(),
        request.title(),
        nextSession,        // 요청 A → sessionNumber = 1
        request.content()   // 요청 B → sessionNumber = 1 (중복!)
);
mentorFeedbackRepository.save(feedback);
mentorship.increaseSession();  // A, B 둘 다 증가 → totalSessions = 2가 되어야 하는데 불일치
```

**2. 회차 이력 데이터 오염**

피드백이 회차별 이력 관리 목적으로 설계되어 있는데, sessionNumber가 중복되면 이력 정렬과 조회가 깨진다.

**3. totalSessions 불일치**

피드백 저장과 세션 증가가 각각 다른 시점에 일어나기 때문에 동시 요청 시 totalSessions 카운트가 실제 피드백 수와 맞지 않게 된다.

## **✅해결 방안**

</aside>

단순히 애플리케이션 레벨에서만 막는 것은 한계가 있다고 판단했다. 네트워크 지연, 트랜잭션 타이밍 등 예외 상황에서 여전히 중복이 발생할 수 있기 때문이다. 이를 해결하기 위해 애플리케이션 레벨과 DB 레벨에서 이중으로 방어하는 전략을 선택했다.

**비관적 락 (애플리케이션 레벨)**

멘토링 row 자체에 락을 걸어 동시 요청을 직렬화한다. 한 요청이 처리되는 동안 다른 요청은 락이 풀릴 때까지 대기하므로 같은 totalSessions 값을 동시에 읽는 상황이 원천 차단된다.

java

```jsx
// MentorshipRepository — 비관적 락 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Mentorship m WHERE m.id = :id")
Optional<Mentorship> findByIdWithLock(@Param("id") Long id);
```

**유니크 제약 (DB 레벨)**

혹시라도 락을 피한 동시 요청이 발생하더라도 DB 레벨에서 차단한다. (mentorship_id, session_number) 조합에 유니크 제약을 걸어 중복 저장 자체를 불가능하게 만든다.

java

```jsx
// MentorFeedback 엔티티 — 유니크 제약 추가
@Table(
    name = "mentorship_feedbacks",
    uniqueConstraints = @UniqueConstraint(
            columnNames = {"mentorship_id", "session_number"}
    )
)
```

**충돌 시 예외 변환**

유니크 제약 충돌 시 DB 예외를 도메인 예외로 변환해 클라이언트에게 명확한 오류 메시지를 반환한다.

java

```jsx
// MentoringServiceV2 — createFeedbackV2()
try {
    mentorFeedbackRepository.save(feedback);
} catch (DataIntegrityViolationException e) {
    // 유니크 제약 충돌 — 동시 요청으로 같은 회차 저장 시도 방어
    throw new ServiceErrorException(MentoringExceptionEnum.MENTORING_SESSION_CONFLICT);
}
```

**♻️ 구현 흐름**

**V1 (기존) — 동시성 보호 없음**

```jsx
요청 A ──→ findById() ──→ totalSessions 읽기(0) ──→ nextSession = 1 ──→ save ──→ increaseSession
요청 B ──→ findById() ──→ totalSessions 읽기(0) ──→ nextSession = 1 ──→ save ──→ increaseSession
                                                              ↑
                                                    둘 다 같은 값 저장 — 중복 발생
```

**V2 (개선) — 비관적 락 + 유니크 제약**

```jsx
요청 A ──→ findByIdWithLock() ──→ 락 획득 ──→ totalSessions 읽기(0) ──→ nextSession = 1
                                                                         ──→ save ──→ increaseSession ──→ 락 해제
요청 B ──→ findByIdWithLock() ──→ 락 대기 ──────────────────────────────────────────────────────────────────────→ 락 획득
                                                                         ──→ totalSessions 읽기(1) ──→ nextSession = 2
                                                                         ──→ save ──→ increaseSession ──→ 락 해제
```

**API 경로 분리**

```jsx
POST /api/v1/mentorings/{mentoringId}/feedbacks  →  MentoringService.createFeedback()       (동시성 보호 없음)
POST /api/v2/mentorings/{mentoringId}/feedbacks  →  MentoringService.createFeedbackV2()     (비관적 락 + 유니크 제약)
```

## 💭원인 분석

</aside>

#### 💥 문제 1 : 단일 노드 `useSingleServer()`

단일 노드 운영이라 Redis 서버가 죽으면 연결된 모든 기능들이 함께 죽는 구조

`RedisConnectionFailureException` 예외가 Redis에 의존되어있는 서버에 전파가 되어 `500 서비스 에러`를 반환되는 구조였다.

```java
// 기존 코드 — 예외 처리 없음
public void setBlackList(String accessToken, Object object, Duration duration) {
    redisTemplate.opsForValue().set(accessToken, object, duration); // Redis 죽으면 바로 예외 전파
}
```

#### 💥 문제 2 : 블랙리스트 데이터 유실

Redis 장애 시 로그아웃 요청이 들어오면 블랙리스트 등록 자체가 **실패**한다.

블랙아웃에 등록될 토큰은 Redis 복구 후에도 블랙리스트에 등록이 실패해 **유효한 토큰으로 인식**이 되어버린다

#### 💥 문제 3 : 랭킹 서비스 버그

지정된 파라미터명은 `realtime` 인데 실제로는 `type`문자열이 그대로 전달되고 있었고, 실제로 Redis가 정상으로 운영되고 있어도 엉뚱한 파라미터 때문에 DB Fallback으로 동작하고 있었다.

```java
// 버그 — type 문자열("realtime")이 Redis key로 들어감
public List<NovelRankingResponse> getNovelRanking(String type) {
    ...
    return buildRankingFromRedis(type, 5); // "realtime" 문자열이 key가 됨
}

public List<NovelRankingResponse> buildRankingFromRedis(String redisKey, int limit) {
    // redisKey = "realtime" → 실제 Redis key인 "ranking:novel:realtime"과 다름
    Set<ZSetOperations.TypedTuple<Object>> topNovels
        = redisTemplate.opsForZSet().reverseRangeWithScores(redisKey, 0, limit - 1);
    // 항상 null 반환 → 항상 DB Fallback으로 빠짐
}
```

Redis가 다운이 되어도 서버가 종료되지 않게하는 싱글 서버에 문제가 생기면 다음 Redis 서버에게 이전과 **데이터 유실 방지**가 필요했다.

## ❓센티넬 선택 이유

</aside>

이미 프로젝트에서 Lua Script와 Redisson 분산락을 사용 중이었고, Cluster는 **샤드**별 분산 저장에 사용하므로 현재 프로젝트에서 사용을 하게 된다면 **오버스팩**이라고 판단했다.

<img width="762" height="1036" alt="Image" src="https://github.com/user-attachments/assets/3ff7bb30-c3e8-499c-8726-aeccbcfa4073" />
- **단일 장애점(SPOF) 제거 및 자동 장애 복구** : 단순 애플리케이션의 DB 우회를 넘어 인프라 자체의 고가용성을 확보하기 위해, 마스터가 다운되면 센티넬(Sentinel)이 이를 감지하고 슬레이브를 마스터로 자동 승격(Failover)시키는 구조가 필요했다.
- **읽기/쓰기 트래픽 분산 (Read/Write Splitting)** : 조회수나 랭킹 확인 등 읽기(Read) 요청이 압도적으로 많은 웹소설 서비스 특성을 반영해 **‘쓰기(Write)’** 는 마스터(Master)로, **‘읽기(Read)’**는 슬레이브(Slave)로 분산시켜 성능의 한계를 극대화할 수 있었고 **‘슬레이브의 복제’** 기능의 기술적 이점이 있었다.

```java
// 장애 대응 흐름 요약

Redis 정상
 	 setBlackList → Redis에 저장
 	 isBlackList  → Redis에서 조회 (빠름)
 	 getRanking   → Redis ZSet에서 조회 (빠름)

Redis Failover 발생 (5~10초 공백)
 	 Sentinel 3대 감지 → 과반수 투표 → Slave 승격
 	 setBlackList 실패 → DB에 저장 (보안 공백 없음)
 	 isBlackList 실패  → DB에서 조회 (차단 유지)
 	 getRanking 실패   → DB 직접 집계 (서비스 유지)

Redis 복구 후
 	 자동으로 Redis 우선 동작 복귀
 	 DB Fallback 데이터는 스케줄러가 정리
```

## ✅해결 과정

</aside>

#### ✅ 1. DB Fallback 적용 로직 (RedisUtil)

센티넬의 Failover(새로운 마스터 선출)가 진행되는 **5~10초의 찰나의 공백기조차 방어하기 위해 애플리케이션 레벨의 DB Fallback 로직**을 구현

예외를 캐치하여 DB에 스냅샷 형태로 임시 저장하고, Redis 복구 후 스케줄러가 이를 정리하도록 설계했다.

```java
 /** === 블랙리스트 등록 - Failover 중 쓰기 실패 시 로그만 남김 === **/
    public void setBlackList(String accessToken, Object object, Duration duration) {
        try {
            redisTemplate.opsForValue().set(accessToken, object, duration);
        } catch (RedisConnectionFailureException | QueryTimeoutException e) {
            log.error("[Redis Blacklist] 등록 실패, accessToken: {}, reason: {}", accessToken, e.getMessage(), e);

            try {    // DB fallback : Redis 장애 중에도 로그아웃 토큰 차단 유지
                if(!blacklistTokenRepository.existsByToken(accessToken)) {
                    LocalDateTime expirationTime = LocalDateTime.now().plus(duration);
                    String reason = (object != null) ? object.toString() : "Unknown";
                    blacklistTokenRepository.save(BlacklistToken.of(accessToken, reason, expirationTime));
                    log.warn("[Redis Blacklist] DB에 블랙리스트 저장 완료, expiredAt: {}", expirationTime);
                }
            } catch (Exception dbEx) {
                log.error("[Redis Blacklist] DB fallback 저장 실패, reason: {}", dbEx.getMessage(), dbEx);
            }
        }
    }
```

#### ✅ 2. **분산 락 호환** 리펙토링 (RedissonUtil)

`LettuceClientConfiguration`의 `ReadFrom.REPLICA_PREFERRED` 옵션을 적용해 읽기/쓰기 분산을 구현했다.

```java
    // Lettuce를 위한 '번역기' 생성 (Redisson의 NatMapper와 동일한 역할)
    @Bean(destroyMethod = "shutdown")
    public ClientResources clientResources() {

        // natMappingIp가 비어있으면 기본 설정 사용 (배포 환경)
        if(natMappingIp == null || natMappingIp.isBlank()) {
            return DefaultClientResources.create();
        }

        // 로컬 환경 - natMappingId (127.0.0.1)로 포워딩
        MappingSocketAddressResolver resolver = MappingSocketAddressResolver.create(
                DnsResolvers.JVM_DEFAULT,
                hostAndPort -> {
                    // 도커 내부 IP(172.x.x.x)를 무시하고 무조건 localhost로 포워딩
                    return HostAndPort.of("127.0.0.1", hostAndPort.getPort());
                }
        );

        return DefaultClientResources.builder()
                .socketAddressResolver(resolver)
                .build();
    }

    /** Redis 다운 시 마스터-슬레이브 및 센티넬 발동
     * 읽기는 슬레이브에서 -> 마스터 부하 분산
     * 슬레이브 장애 시엔 마스터에서 읽음 -> 자동 fallback
     * Failover 새 마스터 선출 시 센티넬이 알려준 새 마스터로 자동 재연결
     **/
    @Bean
    public RedisConnectionFactory redisConnectionFactory(ClientResources clientResources) {
        RedisSentinelConfiguration sentinelConfig = new RedisSentinelConfiguration()
                .master("mymaster")
                .sentinel("127.0.0.1", 26379)
                .sentinel("127.0.0.1", 26380)
                .sentinel("127.0.0.1", 26381);

        LettuceClientConfiguration clientConfig = LettuceClientConfiguration.builder()
                .clientResources(clientResources) // Lettuce 설정에 번역기 장착!
                .readFrom(ReadFrom.REPLICA_PREFERRED)
                .build();

        // 팩토리 생성 후 초기화 (안전성 위해 추가)
        LettuceConnectionFactory factory = new LettuceConnectionFactory(sentinelConfig, clientConfig);
        factory.afterPropertiesSet();
        return factory;
    }
```

기존 단일 서버 기반의 분산 락(Redisson) 설정을 `useSentinelServers()`로 리팩토링하여 장애 상황에서도 Lock 매커니즘이 새 마스터를 동적으로 바라보게 조치했다.

```java
 @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();

        // yml 에서 가져온 노드 주소들 "redis://" 붙여서 배열로 변환
        String[] nodesArray = sentinelNodes.stream()
                .map(node -> "redis://" + node)
                .toArray(String[]::new);

        org.redisson.config.SentinelServersConfig sentinelConfig = config.useSentinelServers()
                .setMasterName(masterName)
                .addSentinelAddress(nodesArray)
                .setCheckSentinelsList(false)  // 3대의 센티넬 교차 검증 통과를 위한 설정 추가
                .setReadMode(org.redisson.config.ReadMode.SLAVE) // 읽기 부하를 Slave로 분산
                .setConnectTimeout(1000)
                .setRetryAttempts(3)
                .setRetryInterval(1500);

        // 배포 시 같은 네트워크 써서 번역기가 필요 없다면 작동하지 않음
        if(natMappingIp != null && !natMappingIp.isBlank()) {
            sentinelConfig.setNatMapper(uri -> new org.redisson.misc.RedisURI(
                    uri.getScheme() + "://" + natMappingIp + ":" + uri.getPort()
            ));
        }
        return Redisson.create(config);
    }
}
```

<summary><b>⚡결제 데이터 정합성 보장 — 타임아웃 시나리오 2건 분석 및 개선</b></summary>

## 배경

서비스는 PortOne(포트원)을 통해 결제 및 환불을 처리한다. 사용자가 결제 또는 취소를 요청하면 서버는 PortOne API를 호출하고, PortOne은 처리 완료 후 웹훅(Webhook)으로 최종 결과를 통지한다.

**문제의 핵심**: 서버가 PortOne API 응답을 기다리는 동안 네트워크 지연·타임아웃이 발생하면, 서버 내부 결제 상태(`FAILED`, `COMPLETED`)와 PortOne 실제 처리 결과 사이에 괴리가 생긴다.

이 괴리를 바로잡을 수 있는 유일한 수단이 웹훅인데, 기존 코드의 `isTerminalStatus()` 로직이 이를 잘못 막고 있었다.

---

## 문제 정의

### 공통 원인: `isTerminalStatus()` 안티패턴

```java
// 기존 코드 (문제)
private boolean isTerminalStatus(PaymentStatus status) {
    return status == PaymentStatus.COMPLETED || status == PaymentStatus.FAILED;
}

// handleWebhook() 내부
if (isTerminalStatus(payment.getStatus())) {
    webhookTransactionService.markEventComplete(webhookEventId);
    return; // ← COMPLETED / FAILED면 무조건 스킵
}
```

`COMPLETED`와 `FAILED`를 "최종 상태"로 간주해 웹훅 처리를 건너뛰었다.

두 가지 타임아웃 시나리오에서 이 로직이 복구를 막는다.

---

### 이슈 1 — `/confirm` 타임아웃: FAILED가 COMPLETED로 복구되지 않음

**시나리오 흐름**

```
사용자 → 결제 요청
  └→ 서버: PortOne /confirm API 호출
       └→ [타임아웃 발생]
            └→ 서버: Payment 상태 = FAILED
                 ← PortOne: 사실은 결제 성공 → Transaction.Paid 웹훅 발송
                      └→ 기존 코드: isTerminalStatus(FAILED) = true → 웹훅 스킵
                           └→ 결과: PortOne 결제 완료 / DB는 FAILED ← 정합성 파괴
```

**영향**

- 사용자는 실제로 돈이 빠져나갔지만 포인트를 받지 못함
- DB 상태 `FAILED` ↔ PortOne 실제 상태 `PAID` 영구 불일치

---

### 이슈 2 — `/cancel` 타임아웃: COMPLETED가 REFUNDED로 전환되지 않음

**시나리오 흐름**

```
사용자 → 환불 요청
  └→ 서버: PortOne 취소 API 호출
       └→ [타임아웃 발생]
            └→ 서버: Payment 상태 = COMPLETED 유지 (취소 결과 미확인)
                 ← PortOne: 사실은 취소 성공 → Transaction.Cancelled 웹훅 발송
                      └→ 기존 코드: isTerminalStatus(COMPLETED) = true → 웹훅 스킵
                           └→ 결과: PortOne 취소 완료 / DB는 COMPLETED ← 정합성 파괴
```

**영향**

- 사용자는 실제로 환불을 받았지만 포인트가 복구되지 않음
- DB 상태 `COMPLETED` ↔ PortOne 실제 상태 `CANCELLED` 영구 불일치

---

## 해결 방안

`isTerminalStatus()` 일괄 분기를 제거하고, **PortOne이 응답한 실제 결제 타입 × DB 상태 조합**으로 명시적 분기를 구성한다.

| DB 상태 | PortOne 웹훅 타입 | 해석 | 처리 |
| --- | --- | --- | --- |
| `FAILED` | `PaidPayment` | confirm 타임아웃 후 결제 성공 확인 | `completePendingPayment()` — 복구 |
| `COMPLETED` | `CancelledPayment` | cancel 타임아웃 후 취소 성공 확인 | `finalizeRefundFromWebhook()` — 복구 |
| `COMPLETED` | `PaidPayment` | 이미 정상 처리됨 | `markEventComplete()` — 스킵 |
| `REFUNDED` | `CancelledPayment` | 이미 환불 완료됨 | `markEventComplete()` — 스킵 |
| `FAILED` | `FailedPayment` | 이미 실패 처리됨 | `markEventComplete()` — 스킵 |

---

## 구현 흐름

### 이슈 1 수정 — FAILED + PaidPayment 보정

```java
if (portOnePayment instanceof PaidPayment paidPayment) {
    if (payment.getStatus() == PaymentStatus.COMPLETED) {
        webhookTransactionService.markEventComplete(webhookEventId);
        return;
    }
    // PENDING 또는 FAILED: 실제 결제 성공 → 복구
    String lockKey = "payment:confirm:lock:" + paymentKey;
    if (!redisUtil.acquireLock(lockKey)) return;
    try {
        webhookTransactionService.completePendingPayment(
            webhookEventId, payment.getId(), toPaymentMethod(paidPayment.getMethod())
        );
    } finally {
        redisUtil.releaseLock(lockKey);
    }
}
```

**변경 포인트**: `FAILED` 상태가 `isTerminalStatus()` 분기를 탈출해 `completePendingPayment()`로 진입. 이 메서드는 `PENDING`과 `FAILED` 둘 다 수용한다.

---

### 이슈 2 수정 — COMPLETED + CancelledPayment 보정

```java
if (portOnePayment instanceof CancelledPayment) {
    if (payment.getStatus() == PaymentStatus.REFUNDED) {
        webhookTransactionService.markEventComplete(webhookEventId);
        return;
    }
    if (payment.getStatus() == PaymentStatus.COMPLETED) {
        // cancel 타임아웃 후 취소 확인 → 복구
        String lockKey = "payment:cancel:lock:" + paymentKey;
        if (!redisUtil.acquireLock(lockKey)) return;
        try {
            webhookTransactionService.finalizeRefundFromWebhook(webhookEventId, payment.getId());
        } finally {
            redisUtil.releaseLock(lockKey);
        }
        return;
    }
    // PENDING → 취소 도착 → FAILED 처리
    webhookTransactionService.failPendingPayment(webhookEventId, payment.getId());
}
```

**변경 포인트**: `COMPLETED` 상태가 `isTerminalStatus()` 분기를 탈출해 `finalizeRefundFromWebhook()`으로 진입. Redis Lock으로 동시 웹훅 재시도에 의한 이중 포인트 복구를 방지한다.

단위 테스트로 두 이슈의 수정 전/후 동작을 검증했다. k6 부하 테스트 대신 단위 테스트를 선택한 이유는, 이 이슈가 **타이밍·상태 조합에 따른 정확성(Correctness)** 문제이기 때문이다. 실제 네트워크 지연은 재현하기

어렵지만, Mock으로 타임아웃 후의 DB 상태를 그대로 재현할 수 있다.

### 테스트 1 — confirm 타임아웃 복구 (`WebhookServiceTest`)

**테스트 목적**

DB 상태가 `FAILED`일 때 `Transaction.Paid` 웹훅이 도착하면 `completePendingPayment()`가 호출되어 COMPLETED로 복구된다.

**테스트 시나리오**

1. DB Payment 상태: `FAILED` (confirm 타임아웃으로 설정됨)
2. PortOne 웹훅: `Transaction.Paid` + `PaidPayment` (실제 결제 성공)
3. Redis Lock 획득: 성공

```java
@Test
@DisplayName("성공 - FAILED Payment를 PaidPayment로 보정 (confirm 타임아웃 케이스)")
void handleWebhook_failedPayment_paidWebhook_completesPayment() {
    // given
    Payment payment = createMockPayment(PAYMENT_ID, PaymentStatus.FAILED);
    PaidPayment paidPayment = mock(PaidPayment.class);
    given(redisUtil.acquireLock(anyString())).willReturn(true);

    // when
    webhookService.handleWebhook(request);

    // then
    verify(webhookTransactionService, times(1))
        .completePendingPayment(eq(WEBHOOK_EVENT_ID), eq(PAYMENT_ID), any(PaymentMethod.class));
    verify(webhookTransactionService, never()).markEventComplete(anyLong());
}
```

**테스트 결과**

- `completePendingPayment()` 1회 호출 확인
- `markEventComplete()` 미호출 확인 (스킵 없이 복구)
- Lock 실패 시 `completePendingPayment()` 미호출 → PortOne 재시도 웹훅 유도

---

### 테스트 2 — cancel 타임아웃 복구 (`WebhookServiceTest`)

**테스트 목적**

DB 상태가 `COMPLETED`일 때 `Transaction.Cancelled` 웹훅이 도착하면 Lock 획득 후 `finalizeRefundFromWebhook()`이 호출되어 REFUNDED로 전환된다.

**테스트 시나리오**

1. DB Payment 상태: `COMPLETED` (cancel 응답을 받지 못한 상태)
2. PortOne 웹훅: `Transaction.Cancelled` + `CancelledPayment`
3. Redis Lock 획득: 성공

```java
@Test
@DisplayName("성공 - COMPLETED Payment + CancelledPayment → Lock 획득 후 finalizeRefundFromWebhook 호출")
void handleWebhook_completedPayment_cancelledWebhook_finalizesRefund() {
    // given
    Payment payment = createMockPayment(PAYMENT_ID, PaymentStatus.COMPLETED);
    given(redisUtil.acquireLock(anyString())).willReturn(true);

    // when
    webhookService.handleWebhook(request);

    // then
    verify(redisUtil, times(1)).acquireLock("payment:cancel:lock:" + PAYMENT_KEY);
    verify(webhookTransactionService, times(1)).finalizeRefundFromWebhook(WEBHOOK_EVENT_ID, PAYMENT_ID);
    verify(redisUtil, times(1)).releaseLock("payment:cancel:lock:" + PAYMENT_KEY);
}
```

**테스트 결과**

- Lock 획득 → 복구 처리 → Lock 해제 순서 확인
- Lock 실패 시 `finalizeRefundFromWebhook()` 미호출 → 이중 포인트 복구 방지
- 예외 발생 시에도 `finally`에서 Lock 해제 확인

### 테스트 3 — 동시 웹훅 경합 (`WebhookConcurrencyTest`)

**테스트 목적**

동일 결제에 대해 웹훅이 동시에 2개 도착해도 `completePendingPayment()`는 정확히 1회만 실행된다.

**테스트 시나리오**

- 스레드 2개가 동시에 동일 결제의 `Transaction.Paid` 웹훅을 처리
- `AtomicBoolean.compareAndSet(false, true)`로 Redis Lock 경합 시뮬레이션

```java
@Test
@DisplayName("동시 웹훅 2개 - 락 경합으로 1개만 completePendingPayment 실행")
void webhook_twoSimultaneousWebhooks_onlyOneCompletesPayment() throws InterruptedException {
    AtomicBoolean lockHeld = new AtomicBoolean(false);
    given(redisUtil.acquireLock(anyString()))
        .willAnswer(inv -> lockHeld.compareAndSet(false, true));

    // 스레드 2개 동시 실행 후
    verify(webhookTransactionService, times(1)).completePendingPayment(anyLong(), anyLong(), any());
}
```

**테스트 결과**

- 2개 스레드 중 Lock을 획득한 1개만 `completePendingPayment()` 실행
- 나머지 1개는 Lock 실패 → 스킵 (PortOne 재시도 위임)
- 예외 없이 정상 완료 (`errorCount = 0`)

---

### 테스트 4 — 예외 발생 시 Lock 해제 보장 (`WebhookConcurrencyTest`)

**테스트 목적**

`completePendingPayment()` 실행 중 예외가 발생해도 `finally` 블록에서 Lock이 반드시 해제된다.

```java
@Test
@DisplayName("completePendingPayment 중 예외 발생해도 finally에서 락 반드시 해제")
void webhook_whenCompletePendingPaymentThrows_lockMustBeReleased() {
    willThrow(new RuntimeException("DB 오류"))
        .given(webhookTransactionService).completePendingPayment(anyLong(), anyLong(), any());

    try { webhookService.handleWebhook(paidWebhookRequest()); } catch (Exception ignored) {}

    verify(redisUtil, times(1)).releaseLock(eq("payment:confirm:lock:" + PORTONE_PAYMENT_ID));
}
```

**테스트 결과**

- 예외 발생 후에도 Lock 해제 1회 확인
- Lock Leak 없음 → 이후 웹훅 재시도 정상 처리 가능

### 해결 효과

- **이슈 1**: confirm 타임아웃 발생 시 PortOne 결제 성공 웹훅이 도착하면 DB를 `FAILED → COMPLETED`로 자동 보정, 포인트 지급 완료

</aside>

멘토링 피드백 작성 API를 구현하면서 피드백의 회차를 자동으로 부여하는 로직이 필요했다. 멘토가 멘티에게 피드백을 작성할 때마다 1회차, 2회차, 3회차... 순서로 이력이 쌓이는 구조로 설계했고, 이를 위해 `Mentorship.totalSessions + 1`을 `sessionNumber`로 사용하는 방식을 채택했다.

CodeRabbit 코드 리뷰 과정에서 이 로직이 동시 요청에 취약하다는 Critical 피드백을 받았고, 실제로 두 요청이 동시에 들어오면 같은 sessionNumber가 중복 저장되어 회차 이력 데이터가 오염될 수 있다는 것을 확인했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 문제 코드 — 동시 요청 시 같은 totalSessions 값을 동시에 읽음
int nextSession = mentorship.getTotalSessions() + 1;

MentorFeedback feedback = MentorFeedback.create(
        mentoringId,
        mentor.getId(),
        request.title(),
        nextSession,        // 요청 A → sessionNumber = 1
        request.content()   // 요청 B → sessionNumber = 1 (중복 저장!)
);
mentorFeedbackRepository.save(feedback);
mentorship.increaseSession();
```

요청 A와 요청 B가 동시에 `getTotalSessions()`를 호출하면 둘 다 0을 읽고 `nextSession = 1`을 계산한다. 결과적으로 sessionNumber가 1인 피드백이 두 개 저장되어 회차 이력이 깨진다. 더불어 `increaseSession()`도 두 번 호출되어 `totalSessions`가 실제 피드백 수와 불일치하게 된다.

<aside>


## 💭가설

</aside>

1. `totalSessions`를 읽고 저장하는 사이에 다른 트랜잭션이 끼어드는 것이 근본 원인이므로, 해당 row를 잠가서 접근을 직렬화하면 해결될 것이다.
2. 애플리케이션 레벨의 락만으로는 네트워크 지연, 트랜잭션 타이밍 등 예외 상황에서 여전히 중복이 발생할 수 있으므로 DB 레벨에서도 이중으로 방어해야 한다.
3. V1은 기존 동작을 유지하고 V2에서 동시성 보호를 적용해 API 경로로 버전을 분리하는 것이 팀원 영향을 최소화하는 방법이다.

## ✅해결 과정

</aside>

**Step 1: 비관적 락 적용 (애플리케이션 레벨)**

멘토링 row 자체에 `PESSIMISTIC_WRITE` 락을 걸어 한 트랜잭션이 처리되는 동안 다른 트랜잭션이 같은 row를 읽지 못하도록 직렬화했다.

java

```jsx
// MentorshipRepository — 비관적 락 메서드 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Mentorship m WHERE m.id = :id")
Optional<Mentorship> findByIdWithLock(@Param("id") Long id);
```

java

```jsx
// MentoringService.createFeedbackV2() — 락 적용
// V2: findByIdWithLock — 비관적 락으로 동시 요청 직렬화
Mentorship mentorship = mentorshipRepository.findByIdWithLock(mentoringId)
        .orElseThrow(() -> new ServiceErrorException(MentoringExceptionEnum.MENTORING_NOT_FOUND));

int nextSession = mentorship.getTotalSessions() + 1;
// 락이 걸린 상태에서 읽으므로 항상 정확한 값을 읽음
```

**Step 2: DB 유니크 제약 추가 (DB 레벨 이중 방어)**

락을 피한 극단적인 동시 요청에도 중복 저장이 불가능하도록 DB 레벨에서 `(mentorship_id, session_number)` 조합에 유니크 제약을 추가했다.

java

```jsx
// MentorFeedback 엔티티 — 유니크 제약 추가
@Table(
    name = "mentorship_feedbacks",
    uniqueConstraints = @UniqueConstraint(
            columnNames = {"mentorship_id", "session_number"}
    )
)
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class MentorFeedback extends BaseEntity {

    @Column(nullable = false)
    private int sessionNumber;   // (mentorship_id, session_number) 유니크 보장

    // ...
}
```

**Step 3: 유니크 제약 충돌 시 예외 변환**

DB 레벨에서 충돌이 발생하더라도 클라이언트에게 500 대신 의미 있는 409 응답을 반환하도록 예외를 변환했다.

java

```jsx
// MentoringExceptionEnum — SESSION_CONFLICT 추가
MENTORING_SESSION_CONFLICT(HttpStatus.CONFLICT,
        "동시 요청으로 인해 피드백 등록에 실패했습니다 잠시 후 다시 시도해 주세요");
```

java

```jsx
// MentoringService.createFeedbackV2() — 예외 변환
try {
    mentorFeedbackRepository.save(feedback);
} catch (DataIntegrityViolationException e) {
    // 유니크 제약 충돌 — DB 예외를 도메인 예외로 변환해 409 반환
    throw new ServiceErrorException(MentoringExceptionEnum.MENTORING_SESSION_CONFLICT);
}
mentorship.increaseSession();
```

**Step 4: V1 / V2 API 경로 분리**

기존 팀원 코드에 영향을 주지 않도록 서비스 단일 클래스 내에서 V1/V2 메서드를 분리하고 컨트롤러에서 API 경로로 버전을 구분했다.

java

```jsx
// V1: 동시성 보호 없음 — 기존 동작 유지
@PostMapping("/api/v1/mentorings/{mentoringId}/feedbacks")
public ResponseEntity<...> createFeedbackV1(...) {
    return mentoringService.createFeedback(mentoringId, userId, request);
}

// V2: 비관적 락 + 유니크 제약 — 동시성 보호
@PostMapping("/api/v2/mentorings/{mentoringId}/feedbacks")
public ResponseEntity<...> createFeedbackV2(...) {
    return mentoringService.createFeedbackV2(mentoringId, userId, request);
}
```

## **💡배경**

웹소설 플랫폼 특성상 메인 페이지의 '신작 소설 목록'은 사용자가 가장 먼저, 그리고 가장 많이 접근하여 트래픽이 집중되는 핵심 기능이라고 생각한다.

초기 구현 시에는 단순 JPA 메서드와 Redis 캐싱만을 활용하여 "최근 30일 이내 생성된 소설"을 조회하도록 기능을 구현해두었던게 발단이었다.

<aside>


## 🚨문제 상황

</aside>

테스트 및 튜터링(멘토링) 과정에서 

> "현재는 데이터가 적어 빠르지만, 서비스가 성장하여 소설 데이터가 수십~수백만 건으로 쌓이게 되면 병목이 발생할 것"
> 

이라는 피드백을 받아 기존 코드를 분석해보니 `is_deleted=false, status='ONGOING', created_at >= 30일` 등의 조회 조건은 걸려 있었으나, **적절한 인덱스를 타지 못해 데이터베이스 전체를 뒤지는 Full Table Scan**이 발생할 위험이 있었다.

<aside>


## 💭가설

</aside>

1. 데이터베이스 계층의 인덱스 부재가 대용량 데이터 조회 시 가장 큰 성능 저하 요인이 될 것이다.
2. 메인 페이지의 신작 목록은 실시간성(초 단위)이 극도로 요구되지 않아 DB 조회 횟수를 최소화하고 캐싱을 적극적으로 활용하면 성능을 안정적으로 유지할 수 있을 것이다.
3. 성능 최적화는 한 번에 오버엔지니어링(예: 샤딩 도입)을 하기보다 `"DB 최적화(인덱싱) → 애플리케이션 최적화(QueryDSL) → 인프라 최적화(Redis 캐싱)"` 순으로 병목 지점을 해결하는 것이 올바른 접근일 것이다.

## ✅해결 과정

</aside>

#### **✅ Step 1: DB 최적화 (복합 인덱스 적용)**

데이터 중복도(카디널리티)와 조회 조건을 고려하여, 조건에 일치하는 데이터만 스캔하도록 Novel 엔티티에 복합 인덱스를 추가했다.

```java
@Table(name = "novels",
        indexes = {
                @Index(name = "idx_novel_genre", columnList = "genre"),
                @Index(name = "idx_novel_status", columnList = "status"),

                // 신작 소설 조회용 복합 인덱스
                @Index(name = "idx_novel_new_list", columnList = "is_deleted, status, genre, created_at")
        })
public class Novel extends BaseEntity {
```

#### **✅ Step 2: 캐싱 전략 고도화 (Look-Aside 패턴)**

- 실시간성에 목매지 않아도 되는 신작 리스트의 특성을 살려, 조회 시 Redis를 먼저 확인(Cache Hit)하고 데이터가 없을 때만 DB를 조회(Cache Miss)하는 Look-Aside 패턴을 구현
- 장르, 상태, 제한 개수(limit)를 조합한 Cache Key(`NOVEL_LIST_CACHE_KEY + "type:new::" + "genre:" + ...`)를 설계하고 적절한 TTL(30분)을 부여하여 무의미한 DB 접근을 차단했다.

```java
// 신작용 소설 목록 조회 (QueryDSL+Redis캐싱+인덱싱)
    // 한 달 신작 리스트 (limit 50 작품)
    @Transactional(readOnly = true)
    public List<NovelListResponse> getNewNovelList(String genre, NovelStatus status, int limit) {

        String cacheKey = NOVEL_LIST_CACHE_KEY
                + "type:new::"
                + "genre:" + genre + "::"
                + "status:" + status + "::"
                + "limit:" + limit;

        // Redis 캐시 확인
        Object cached = null;
        try {
            cached = redisTemplate.opsForValue().get(cacheKey);
        } catch (RuntimeException e) {
            log.warn("소설 리스트 캐시를 읽어내는데 실패했습니다. key={}", cacheKey, e);
        }

        if (cached != null) {
            log.debug("===== [신작 캐시 HIT] key={} =====", cacheKey);
            return (List<NovelListResponse>) cached; // List 타입 캐스팅
        }

        log.debug("===== [신작 캐시 MISS] key={} DB 조회 =====", cacheKey);

        // 3. DB 조회 (캐시에 없을 때만)
        List<NovelListResponse> response = novelRepository.findNewNovelList(genre, status, limit);

        // 4. Redis 캐시에 결과 저장
        try {
            redisTemplate.opsForValue().set(cacheKey, response, NOVEL_LIST_CACHE_TTL);
            log.debug("===== [신작 캐시 저장] key={} TTL=30분 =====", cacheKey);
        } catch (RuntimeException e) {
            log.warn("New novel list cache write failed. key={}", cacheKey, e);
        }

        return response;
      }
```

#### **✅ Step 3: QueryDSL을 통한 쿼리 최적화**

QueryDSL의 `Projections.constructor`를 활용하여 엔티티 전체가 아닌 DTO 형태로 꼭 필요한 필드만 Select 하도록 구성하고 Fetch Join을 통해 N+1 문제를 방지했다.

```java
// 신작용 소설 목록 조회 (한 달 신작 리스트)
    @Override
    public List<NovelListResponse> findNewNovelList(String genre, NovelStatus status, int limit) {
        QNovel novel = QNovel.novel;
        QUser user = QUser.user;

        // 한 달 전 날짜 계산
        LocalDateTime oneMonthAgo = LocalDateTime.now().minusMonths(1);

        return queryFactory
                .select(Projections.constructor(NovelListResponse.class,
                        novel.id,
                        novel.title,
                        novel.genre,
                        novel.tags,
                        novel.status,
                        novel.coverImageUrl,
                        novel.viewCount,
                        novel.bookmarkCount,
                        user.nickname
                        ))
                .from(novel)
                .join(user).on(novel.authorId.eq(user.id))
                .where(
                        novel.isDeleted.eq(false)
                        , novel.status.ne(NovelStatus.PENDING)
                        , genreEq(genre)
                        , statusEq(status)
                        , novel.createdAt.goe(oneMonthAgo) // 한 달 이내 신작 소설들
                )
                .orderBy(novel.createdAt.desc())
                .limit(limit)
                .fetch();

    }
```

## ✅배운점

</aside>

Full Table Scan으로 인해 발생할 수 있었던 메인 페이지 병목 현상을 복합 인덱스와 QueryDSL을 통해 안전하게 방어하면서 

Redis 캐시 히트율을 높임으로써 메인 페이지에 트래픽이 몰리더라도 DB에 직접적인 부하가 가지 않는 안정적인 조회 아키텍처를 구축해볼 수 있었다.

그리고 무작정 파티셔닝이나 샤딩 같은 고도의 기술을 맹목적으로 도입하기보다는 **처음부터 천천히 현재 트래픽 규모에 맞는 기본기(Index + Cache)**부터 단계적으로 성능을 끌어올리는 인프라 설계를 머릿속에 새길 수 있었다.

- **악성 요청 방어**

    - 도구 : k6
    - maxDuration : 10s
    - 성공 기준 :
        - SMS 정상 발송 200 및 5회 인증 초과 등

### ✍️ 시나리오 B

- **동시 회원가입 플로우**
    - 동시 다발로 각기 다른 번호로 회원가입 SMS 승인 요청을 보낸다.
- **사용자 수 가정**
    - 500명 / 1000명 / 5000명 / 10,000명 으로 나눠 테스트
- **테스트 조건**
    - 도구 : k6
    - maxDuration : 1m
    - 성공 기준 :
        - 1단계 SMS 전송 성공
        - 2단계 SMS 인증 성공
        - 3단계 공통 회원가입 성공
        - 4단계 작가/독자 회원가입 최종 등록 성공

### 📌  플로우 흐름

SMS 인증번호 송신 → 인증번호 확인(임시 토큰 발급) → 1단계 공통 회원가입(Redis 임시 토큰) → 2단계 작가/독자 가입(최종 DB Insert)가입

이 때 발생하는 발생하는 DB의 Unique 제약조건(중복 예외) 오류로 인한 가짜 성능 결과를 막기 위해 각 가상 유저마다 이메일, 닉네임, 폰번호를 동적으로 다르게 생성하여 실제와 동일한 쓰기(Insert) 부하를 유발하도록 구성

## **📚배경**

</aside>

회차 본문 조회 성능을 개선하기 위해 Redis 벌크 캐싱(V2)을 도입했다. 50화 단위로 묶어서 캐싱하면 한 번에 여러 회차를 Redis에서 가져올 수 있어 DB 부하가 줄어들 것이라 판단했다.

하지만 K6 부하테스트 결과가 예상과 완전히 반대로 나왔다. 캐시 히트율이 100%에 가까운데도 V2가 V1보다 오히려 훨씬 더 느렸다.

<aside>


## 🚨문제 정의

</aside>

### 1. 벌크 캐싱 도입 후 오히려 성능 저하

| 지표 | V1 (JPA 직접 조회) | V2 (벌크 캐싱) |
| --- | --- | --- |
| 평균 응답시간 | 9ms | 195ms |
| 캐시 히트율 | - | 100% |

캐시를 쓰는데 오히려 **21배** 더 느린 황당한 결과가 나왔다.

### 2. 원인 분석

기존 벌크 캐싱 구조:

```
episode_bulk::1::1 키에 50개 회차 통째로 저장 (약 20만자 JSON)
```

문제는 회차 1개만 필요한데 매 요청마다 50개 회차 전체를 역직렬화해야 한다는 것이었다.

```
[V1 흐름]
DB SELECT (PK 인덱스, 1ms) → 응답

[V2 흐름]
Redis GET (50개 묶음, 20만자) → JSON 역직렬화 (50배 비용) → 1개 추출 → 응답
```

- 회차 1개 응답하기 위해 50배의 역직렬화 비용 발생
- Redis → 애플리케이션 간 불필요한 대용량 데이터 전송
- 캐시 히트율 100%여도 역직렬화 오버헤드가 DB 조회 비용보다 훨씬 큼

## ✅해결 방안

</aside>

벌크 캐싱에서 **단건 캐싱**으로 전환

```java
// 기존: 벌크 캐싱 (50개 묶음)
public List<EpisodeBulkCache> getBulkCache(Long novelId, int bulkIndex) {
    // 50개 회차 통째로 가져와서 역직렬화
}

// 변경: 단건 캐싱 (1개씩)
public EpisodeContentCache getContentCache(Long episodeId) {
    // 1개 회차만 가져와서 역직렬화
}
```

**캐시 키 변경:**

```
기존: episode_bulk::1::1  (50개 묶음)
변경: episode_content::1  (1개씩)
```

추가로 모든 회차를 캐싱하면 Redis 메모리가 무한정 늘어나는 문제가 있어서, **핫키 임계값(1시간 내 50회 이상 조회)을 넘긴 인기 소설 회차만 선별 캐싱**하여 메모리 효율도 확보했다.

## **💡배경**

</aside>

멘토링 도메인 초기 설계 시 `Mentorship` 엔티티에 `title` 필드를 추가했다. 멘토링 신청 시 제목을 입력받아 저장하는 구조였고, 당시에는 자연스러운 설계처럼 보였다.

코드 리뷰 과정에서 팀원이 `title`을 `MentorFeedback`으로 옮겨야 한다는 의견을 제시했다. 처음에는 `Mentorship.title`이 멘토링 신청 자체를 식별하는 제목이므로 현재 위치가 맞다고 판단했지만, 실제 도메인 구조와 피그마 화면을 다시 분석하면서 설계 오류를 발견했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 기존 Mentorship 엔티티 — title이 신청 단위에 묶여있음
public static Mentorship create(Long mentorId, Long menteeId, Long currentNovelId,
                                String motivation, String manuscriptUrl, String title) {
    Mentorship mentorship = new Mentorship();
    mentorship.title = title;       // 멘토링 신청 전체의 제목
    mentorship.totalSessions = 0;   // 회차는 숫자만 있고 내용이 없음
    // ...
}

// 기존 MentorFeedback 엔티티 — 피드백 내용만 있고 회차 식별 불가
public static MentorFeedback create(Long mentorshipId, Long authorId, String content) {
    return new MentorFeedback(mentorshipId, authorId, content);
    // 몇 회차인지, 이 회차의 제목이 무엇인지 알 수 없음
}
```

`totalSessions`는 숫자 카운트만 있고 각 회차의 내용이 `MentorFeedback`에 담겨있지 않아 이력 관리가 불완전한 구조였다. 멘토 입장에서 최대 5명의 멘티를 동시에 관리하는 1:1 구조에서, 각 멘토링의 회차별 피드백 제목이 없으면 이력을 구분할 방법이 없었다.

## 💭가설

</aside>

1. `title`이 표현하는 것이 **"멘토링 신청 전체의 제목"** 인지, **"각 피드백 회차의 제목"** 인지를 명확히 정의하면 올바른 위치가 결정될 것이다.
2. 피그마 화면에서 실제로 어떤 데이터가 표시되는지 확인하면 설계 방향이 명확해질 것이다.
3. `Mentorship.title` 제거 후 상세 조회에서 제목이 필요한 경우 `Novel.title`로 대체할 수 있을 것이다.

**Step 1: 도메인 책임 분리 원칙으로 위치 재정의**

피그마 화면 분석 결과, 멘토링 상세 화면에서 표시되는 것은 소설 제목이었고 별도의 멘토링 신청 제목은 존재하지 않았다. 각 피드백 회차를 구분하는 제목이 필요한 것이었으므로 `title`은 `MentorFeedback`에 있어야 한다는 결론을 내렸다.


<summary><b>📌웹훅 미수신으로 인한 결제 정합성 영구 불일치</b></summary>

## 📌 배경

본 프로젝트는 포트원(PortOne) V2를 연동한 포인트 충전 결제 시스템을 운영 중이다.

결제 흐름은 아래와 같다.

1. 프론트에서 포트원 SDK를 통해 결제 시작 → 우리 DB에 `PENDING` 상태 기록
2. 결제 완료 후 프론트가 `/confirm` 호출 → 포트원 상태 검증 후 `COMPLETED` 처리 및 포인트 지급
3. 포트원이 웹훅을 비동기로 전송 → 결제 상태를 재검증하여 불일치 보정

이 구조에서 웹훅은 상태 보완 수단으로 설계되어 있었다.

---

## ❗ 문제 상황

다음과 같은 시나리오에서 웹훅이 아예 도달하지 않거나, 처리에 실패할 수 있다.

- **네트워크 장애**: 포트원 서버와 우리 서버 사이의 일시적 단절
- **서버 다운타임**: 웹훅 수신 시점에 우리 서버가 재배포 중이거나 일시 장애 상태
- **재시도 소진**: 포트원의 재시도 정책(수회)을 모두 소진한 뒤에도 수신 실패

이 경우 포트원에서는 결제가 `PAID`로 처리됐지만 우리 DB는 `PENDING` 또는 `FAILED` 상태로 남아

**영구적인 정합성 불일치**가 발생한다.

사용자는 실제로 결제에 성공했음에도 포인트를 받지 못하는 치명적 상황이 생길 수 있다.

---

## 💡 가설

### 가설 1 — 웹훅만으로는 정합성을 보장할 수 없다

웹훅은 포트원이 "전송을 시도"하는 이벤트다. 수신 보장을 전제로 설계하면 단일 실패 지점이 된다.

외부 의존성이 있는 이상, 웹훅이 유실되는 케이스를 별도로 방어해야 한다.

### 가설 2 — 주기적 폴링으로 유실된 이벤트를 사후 보정할 수 있다

포트원 API는 `paymentKey`로 현재 결제 상태를 직접 조회할 수 있다.

일정 시간이 지나도록 상태가 갱신되지 않은 건들을 주기적으로 포트원에 직접 조회하면

웹훅 유실 여부와 무관하게 최종 상태로 보정할 수 있다.

---

## 🛠 해결 과정

### 1단계 — 보정 대상 기준 설계

단순히 "오래된 PENDING을 전부 조회"하면 정상 진행 중인 결제까지 건드릴 수 있다.

아래 두 가지 케이스를 보정 대상으로 정의했다.

| 대상 | 조건 | 사유 |
| --- | --- | --- |
| `PENDING` 30분 초과 | `createdAt < now - 30m` | `/confirm` 미호출 + 웹훅 유실이 동시에 발생한 케이스 |
| `FAILED` 30분~2시간 이내 | `createdAt between now-2h and now-30m` | confirm 타임아웃으로 잘못 FAILED 처리 후 웹훅도 유실된 케이스 |

> **30분 하한선을 둔 이유**: 방금 실패한 건은 정상적인 FAILED일 가능성이 높다.
> 
> 
> 30분이 지난 FAILED는 그 사이 웹훅이 도달했다면 이미 처리됐을 것이므로,
> 
> 미처리 상태로 남아있다면 웹훅 유실로 판단할 수 있다.
> 

### 2단계 — PaymentReconciliationService 구현

```java
@Scheduled(fixedDelay = 10, timeUnit = TimeUnit.MINUTES)
public void reconcile() {
    LocalDateTime now = LocalDateTime.now();

    // PENDING이 30분 넘도록 그대로인 건 — /confirm 미호출 + 웹훅 유실
    List<Payment> stalePending = paymentRepository
            .findByStatusAndCreatedAtBefore(PaymentStatus.PENDING, now.minusMinutes(30));

    // FAILED 30분~2시간 이내인 건 — confirm 타임아웃으로 잘못 FAILED 처리된 가능성
    List<Payment> recentFailed = paymentRepository
            .findByStatusAndCreatedAtBetween(PaymentStatus.FAILED, now.minusHours(2), now.minusMinutes(30));

    stalePending.forEach(p -> reconcileOne(p, "PENDING"));
    recentFailed.forEach(p -> reconcileOne(p, "FAILED"));
}
```

### 3단계 — 포트원 직접 조회 후 상태 보정

각 건에 대해 포트원 API를 직접 조회하여 실제 결제 상태를 기준으로 보정한다.

```java
private void reconcileOne(Payment payment, String reason) {
    io.portone.sdk.server.payment.Payment portOnePayment =
        paymentClient.getPayment(payment.getPaymentKey()).get(10, TimeUnit.SECONDS);

    if (portOnePayment instanceof PaidPayment paidPayment) {
        // 포트원은 성공, 우리는 PENDING/FAILED → COMPLETED + 포인트 충전
        paymentTransactionService.completePayment(...);

    } else if (portOnePayment instanceof FailedPayment || portOnePayment instanceof CancelledPayment) {
        // 포트원도 실패/취소 → FAILED로 정리
        paymentTransactionService.failPayment(payment.getId());

    } else {
        // 포트원도 아직 PENDING (진행 중) → 스킵
    }
}
```

### 최종 방어 구조

```
결제 실패/타임아웃 발생
│
├─ 1단계: cancelPayment() 내 compensateDeduct() ─── 즉시 복구 (포트원 상태 불명 시 보상)
│
├─ 2단계: 포트원 웹훅 도착 ────────────────────────── 수초~수십 초 내 보정
│         ├─ CancelledPayment + COMPLETED → REFUNDED + 포인트 처리
│         └─ PaidPayment + FAILED → COMPLETED + 포인트 충전
│
└─ 3단계: 재검증 배치 (10분 주기) ────────────────── 웹훅 유실 시 최후 방어선
          └─ 포트원 직접 조회 → 상태 보정
```

웹훅은 **"보완 수단"**, 재검증 배치가 **"보장 수단"** 이 된다.

---

## ✅ 결과

- `PENDING` / `FAILED` 상태로 방치되던 결제 건을 10분 이내에 자동으로 정합성 복구
- 웹훅 유실, 네트워크 장애, 서버 다운타임 등 모든 외부 요인에 무관하게 최종 일관성 보장
- 사용자가 결제 성공 후 포인트를 받지 못하는 케이스 제거

## 📚 배운 점

### 1. 외부 이벤트(웹훅)는 "최선 노력(best-effort)" 전달이다

외부 시스템이 전송하는 이벤트는 항상 유실 가능성이 있다.

이를 유일한 상태 동기화 수단으로 삼으면 단일 실패 지점이 된다.
웹훅은 빠른 반응을 위한 보완 수단이고, 주기적 폴링이 정합성을 **보장**하는 수단이어야 한다.

### 2. 재시도와 멱등성은 세트다

배치에서 동일한 결제 건을 여러 번 처리할 수 있으므로 `completePayment`, `failPayment` 등

모든 상태 전환 로직은 **멱등하게** 설계되어야 한다.
이미 `COMPLETED`인 건에 다시 포인트를 지급하지 않도록 상태 검사를 선행해야 한다.

### 3. 보정 대상의 시간 범위 설계가 핵심이다

"언제부터 언제까지"를 잘못 설정하면 정상 건을 건드리거나 실패 건을 놓칠 수 있다.

시스템의 응답 시간, 포트원 재시도 주기, 정상 결제 소요 시간을 함께 고려해서

하한/상한 기준을 신중하게 정해야 한다.

### 4. 결제 시스템은 3단계 방어를 기본으로 설계하라

즉시 보상(compensate) → 비동기 이벤트(webhook) → 주기 배치(reconciliation)

이 세 레이어가 갖춰져야 외부 의존성이 있는 결제 시스템에서 데이터 정합성을 신뢰할 수 있다.
