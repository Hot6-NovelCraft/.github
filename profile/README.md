### 2. 프로젝트 개요
*개발 기간 : 2025.04.06 ~ 2025.05.15

NovelCraft 는 독자와 작가 모두를 위한 웹소설 양성 플랫폼입니다.

단순한 소설 감상 서비스를 넘어 신입 작가의 성장과 
독자의 독서 습관 형성을 동시에 지원하는 것이 핵심 목표입니다.

또한 성장하는 웹소설 시장에서 신입 작가들이 데뷔하고 성장할 수 있는 
체계적인 발판을 지원하기 위해 멘토/멘티 시스템을 도입, 
경력 작가와 신입 작가를 연결, 
독자에게는 다양한 독서 경험을 제공합니다.

### 10. 트러블 슈팅
<details>
<summary><b>전민우 - 멘토링 피드백 회차 동시성 이슈 — sessionNumber 중복 저장 방어</b></summary>

<br>

<aside>


## **💡배경**

</aside>

멘토링 피드백 작성 API를 구현하면서 피드백의 회차를 자동으로 부여하는 로직이 필요했다. 멘토가 멘티에게 피드백을 작성할 때마다 1회차, 2회차, 3회차... 순서로 이력이 쌓이는 구조로 설계했고, 이를 위해 `Mentorship.totalSessions + 1`을 `sessionNumber`로 사용하는 방식을 채택했다.

CodeRabbit 코드 리뷰 과정에서 이 로직이 동시 요청에 취약하다는 Critical 피드백을 받았고, 실제로 두 요청이 동시에 들어오면 같은 sessionNumber가 중복 저장되어 회차 이력 데이터가 오염될 수 있다는 것을 확인했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 문제 코드 — 동시 요청 시 같은 totalSessions 값을 동시에 읽음
int nextSession = mentorship.getTotalSessions() + 1;

MentorFeedback feedback = MentorFeedback.create(
        mentoringId,
        mentor.getId(),
        request.title(),
        nextSession,        // 요청 A → sessionNumber = 1
        request.content()   // 요청 B → sessionNumber = 1 (중복 저장!)
);
mentorFeedbackRepository.save(feedback);
mentorship.increaseSession();
```

요청 A와 요청 B가 동시에 `getTotalSessions()`를 호출하면 둘 다 0을 읽고 `nextSession = 1`을 계산한다. 결과적으로 sessionNumber가 1인 피드백이 두 개 저장되어 회차 이력이 깨진다. 더불어 `increaseSession()`도 두 번 호출되어 `totalSessions`가 실제 피드백 수와 불일치하게 된다.

---

<aside>


## 💭가설

</aside>

1. `totalSessions`를 읽고 저장하는 사이에 다른 트랜잭션이 끼어드는 것이 근본 원인이므로, 해당 row를 잠가서 접근을 직렬화하면 해결될 것이다.
2. 애플리케이션 레벨의 락만으로는 네트워크 지연, 트랜잭션 타이밍 등 예외 상황에서 여전히 중복이 발생할 수 있으므로 DB 레벨에서도 이중으로 방어해야 한다.
3. V1은 기존 동작을 유지하고 V2에서 동시성 보호를 적용해 API 경로로 버전을 분리하는 것이 팀원 영향을 최소화하는 방법이다.

---

<aside>


## ✅해결 과정

</aside>

**Step 1: 비관적 락 적용 (애플리케이션 레벨)**

멘토링 row 자체에 `PESSIMISTIC_WRITE` 락을 걸어 한 트랜잭션이 처리되는 동안 다른 트랜잭션이 같은 row를 읽지 못하도록 직렬화했다.

java

```jsx
// MentorshipRepository — 비관적 락 메서드 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Mentorship m WHERE m.id = :id")
Optional<Mentorship> findByIdWithLock(@Param("id") Long id);
```

java

```jsx
// MentoringService.createFeedbackV2() — 락 적용
// V2: findByIdWithLock — 비관적 락으로 동시 요청 직렬화
Mentorship mentorship = mentorshipRepository.findByIdWithLock(mentoringId)
        .orElseThrow(() -> new ServiceErrorException(MentoringExceptionEnum.MENTORING_NOT_FOUND));

int nextSession = mentorship.getTotalSessions() + 1;
// 락이 걸린 상태에서 읽으므로 항상 정확한 값을 읽음
```

**Step 2: DB 유니크 제약 추가 (DB 레벨 이중 방어)**

락을 피한 극단적인 동시 요청에도 중복 저장이 불가능하도록 DB 레벨에서 `(mentorship_id, session_number)` 조합에 유니크 제약을 추가했다.

java

```jsx
// MentorFeedback 엔티티 — 유니크 제약 추가
@Table(
    name = "mentorship_feedbacks",
    uniqueConstraints = @UniqueConstraint(
            columnNames = {"mentorship_id", "session_number"}
    )
)
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class MentorFeedback extends BaseEntity {

    @Column(nullable = false)
    private int sessionNumber;   // (mentorship_id, session_number) 유니크 보장

    // ...
}
```

**Step 3: 유니크 제약 충돌 시 예외 변환**

DB 레벨에서 충돌이 발생하더라도 클라이언트에게 500 대신 의미 있는 409 응답을 반환하도록 예외를 변환했다.

java

```jsx
// MentoringExceptionEnum — SESSION_CONFLICT 추가
MENTORING_SESSION_CONFLICT(HttpStatus.CONFLICT,
        "동시 요청으로 인해 피드백 등록에 실패했습니다 잠시 후 다시 시도해 주세요");
```

java

```jsx
// MentoringService.createFeedbackV2() — 예외 변환
try {
    mentorFeedbackRepository.save(feedback);
} catch (DataIntegrityViolationException e) {
    // 유니크 제약 충돌 — DB 예외를 도메인 예외로 변환해 409 반환
    throw new ServiceErrorException(MentoringExceptionEnum.MENTORING_SESSION_CONFLICT);
}
mentorship.increaseSession();
```

**Step 4: V1 / V2 API 경로 분리**

기존 팀원 코드에 영향을 주지 않도록 서비스 단일 클래스 내에서 V1/V2 메서드를 분리하고 컨트롤러에서 API 경로로 버전을 구분했다.

java

```jsx
// V1: 동시성 보호 없음 — 기존 동작 유지
@PostMapping("/api/v1/mentorings/{mentoringId}/feedbacks")
```

## **💡배경**

</aside>

웹소설 플랫폼 특성상 메인 페이지의 '신작 소설 목록'은 사용자가 가장 먼저, 그리고 가장 많이 접근하여 트래픽이 집중되는 핵심 기능이라고 생각한다.

초기 구현 시에는 단순 JPA 메서드와 Redis 캐싱만을 활용하여 "최근 30일 이내 생성된 소설"을 조회하도록 기능을 구현해두었던게 발단이었다.

<aside>


## 🚨문제 상황

</aside>

테스트 및 튜터링(멘토링) 과정에서 

> "현재는 데이터가 적어 빠르지만, 서비스가 성장하여 소설 데이터가 수십~수백만 건으로 쌓이게 되면 병목이 발생할 것"
> 

이라는 피드백을 받아 기존 코드를 분석해보니 `is_deleted=false, status='ONGOING', created_at >= 30일` 등의 조회 조건은 걸려 있었으나, **적절한 인덱스를 타지 못해 데이터베이스 전체를 뒤지는 Full Table Scan**이 발생할 위험이 있었다.

<aside>


## 💭가설

</aside>

1. 데이터베이스 계층의 인덱스 부재가 대용량 데이터 조회 시 가장 큰 성능 저하 요인이 될 것이다.
2. 메인 페이지의 신작 목록은 실시간성(초 단위)이 극도로 요구되지 않아 DB 조회 횟수를 최소화하고 캐싱을 적극적으로 활용하면 성능을 안정적으로 유지할 수 있을 것이다.
3. 성능 최적화는 한 번에 오버엔지니어링(예: 샤딩 도입)을 하기보다 `"DB 최적화(인덱싱) → 애플리케이션 최적화(QueryDSL) → 인프라 최적화(Redis 캐싱)"` 순으로 병목 지점을 해결하는 것이 올바른 접근일 것이다.

## ✅해결 과정

</aside>

#### **✅ Step 1: DB 최적화 (복합 인덱스 적용)**

데이터 중복도(카디널리티)와 조회 조건을 고려하여, 조건에 일치하는 데이터만 스캔하도록 Novel 엔티티에 복합 인덱스를 추가했다.

```java
@Table(name = "novels",
        indexes = {
                @Index(name = "idx_novel_genre", columnList = "genre"),
                @Index(name = "idx_novel_status", columnList = "status"),

                // 신작 소설 조회용 복합 인덱스
                @Index(name = "idx_novel_new_list", columnList = "is_deleted, status, genre, created_at")
        })
public class Novel extends BaseEntity {
```

#### **✅ Step 2: 캐싱 전략 고도화 (Look-Aside 패턴)**

- 실시간성에 목매지 않아도 되는 신작 리스트의 특성을 살려, 조회 시 Redis를 먼저 확인(Cache Hit)하고 데이터가 없을 때만 DB를 조회(Cache Miss)하는 Look-Aside 패턴을 구현
- 장르, 상태, 제한 개수(limit)를 조합한 Cache Key(`NOVEL_LIST_CACHE_KEY + "type:new::" + "genre:" + ...`)를 설계하고 적절한 TTL(30분)을 부여하여 무의미한 DB 접근을 차단했다.

```java
// 신작용 소설 목록 조회 (QueryDSL+Redis캐싱+인덱싱)
    // 한 달 신작 리스트 (limit 50 작품)
    @Transactional(readOnly = true)
    public List<NovelListResponse> getNewNovelList(String genre, NovelStatus status, int limit) {

        String cacheKey = NOVEL_LIST_CACHE_KEY
                + "type:new::"
                + "genre:" + genre + "::"
                + "status:" + status + "::"
                + "limit:" + limit;

        // Redis 캐시 확인
        Object cached = null;
        try {
            cached = redisTemplate.opsForValue().get(cacheKey);
        } catch (RuntimeException e) {
            log.warn("소설 리스트 캐시를 읽어내는데 실패했습니다. key={}", cacheKey, e);
        }

        if (cached != null) {
            log.debug("===== [신작 캐시 HIT] key={} =====", cacheKey);
            return (List<NovelListResponse>) cached; // List 타입 캐스팅
        }

        log.debug("===== [신작 캐시 MISS] key={} DB 조회 =====", cacheKey);

        // 3. DB 조회 (캐시에 없을 때만)
        List<NovelListResponse> response = novelRepository.findNewNovelList(genre, status, limit);

        // 4. Redis 캐시에 결과 저장
        try {
            redisTemplate.opsForValue().set(cacheKey, response, NOVEL_LIST_CACHE_TTL);
            log.debug("===== [신작 캐시 저장] key={} TTL=30분 =====", cacheKey);
        } catch (RuntimeException e) {
            log.warn("New novel list cache write failed. key={}", cacheKey, e);
        }

        return response;
      }
```

#### **✅ Step 3: QueryDSL을 통한 쿼리 최적화**

QueryDSL의 `Projections.constructor`를 활용하여 엔티티 전체가 아닌 DTO 형태로 꼭 필요한 필드만 Select 하도록 구성하고 Fetch Join을 통해 N+1 문제를 방지했다.

```java
// 신작용 소설 목록 조회 (한 달 신작 리스트)
    @Override
    public List<NovelListResponse> findNewNovelList(String genre, NovelStatus status, int limit) {
        QNovel novel = QNovel.novel;
        QUser user = QUser.user;

        // 한 달 전 날짜 계산
        LocalDateTime oneMonthAgo = LocalDateTime.now().minusMonths(1);

        return queryFactory
                .select(Projections.constructor(NovelListResponse.class,
                        novel.id,
                        novel.title,
                        novel.genre,
                        novel.tags,
                        novel.status,
                        novel.coverImageUrl,
                        novel.viewCount,
                        novel.bookmarkCount,
                        user.nickname
                        ))
                .from(novel)
                .join(user).on(novel.authorId.eq(user.id))
                .where(
                        novel.isDeleted.eq(false)
                        , novel.status.ne(NovelStatus.PENDING)
                        , genreEq(genre)
                        , statusEq(status)
                        , novel.createdAt.goe(oneMonthAgo) // 한 달 이내 신작 소설들
                )
                .orderBy(novel.createdAt.desc())
                .limit(limit)
                .fetch();

    }
```

Full Table Scan으로 인해 발생할 수 있었던 메인 페이지 병목 현상을 복합 인덱스와 QueryDSL을 통해 안전하게 방어하면서

그리고 무작정 파티셔닝이나 샤딩 같은 고도의 기술을 맹목적으로 도입하기보다는 **처음부터 천천히 현재 트래픽 규모에 맞는 기본기(Index + Cache)**부터 단계적으로 성능을 끌어올리는 인프라 설계를 머릿속에 새길 수 있었다.

## 📊최종 정리

</aside>

- **시나리오 A - 악성 요청 방어 (단일 핸드폰 번호)**
    - **Redis 원자성 보장 성공** 
    단순 인증 로직에 무거운 분산 락을 거는 대신 싱글 스레드로 동작하는 Redis의 특징을 활용한 Lua Script로 동시성 취약점을 완벽히 해결하고 불필요한 SMS 과금 방어

분산 락 대신 오버헤드가 적은 **Redis Lua Script**를 도입하여 GET과 DEL 명령어를 묶어 원자적(Atomic)으로 처리했다.

그 결과 500개의 동시 검증 요청 중 단 1개의 요청만 정상 처리(200 OK)되고, 나머지 499개의 요청은 인증번호가 이미 삭제되어 실패(4xx 에러) 처리됨을 확인했다.

- **시나리오 B - 동시 회원가입 플로우**
    - **성능 테스트 고도화 경험** 
    단순 API 테스트에 그치지 않고 k6를 통한 부하 테스트를 직접 스크립팅해 정합성 보장뿐만 아니라 시스템 한계(스레드 풀, 병목 지점)를 시각적으로(Grafana 대시보드) 모니터링하고 파악하는 실무적인 성능 검증

<summary><b>정은식 - 벌크 캐싱 도입 후 성능 저하 → 단건 캐싱으로 전환</b></summary>
<aside>


## **📚배경**

</aside>

회차 본문 조회 성능을 개선하기 위해 Redis 벌크 캐싱(V2)을 도입했다. 50화 단위로 묶어서 캐싱하면 한 번에 여러 회차를 Redis에서 가져올 수 있어 DB 부하가 줄어들 것이라 판단했다.

하지만 K6 부하테스트 결과가 예상과 완전히 반대로 나왔다. 캐시 히트율이 100%에 가까운데도 V2가 V1보다 오히려 훨씬 더 느렸다.

<aside>


## 🚨문제 정의

</aside>

### 1. 벌크 캐싱 도입 후 오히려 성능 저하

| 지표 | V1 (JPA 직접 조회) | V2 (벌크 캐싱) |
| --- | --- | --- |
| 평균 응답시간 | 9ms | 195ms |
| 캐시 히트율 | - | 100% |

캐시를 쓰는데 오히려 **21배** 더 느린 황당한 결과가 나왔다.

### 2. 원인 분석

기존 벌크 캐싱 구조:

```
episode_bulk::1::1 키에 50개 회차 통째로 저장 (약 20만자 JSON)
```

문제는 회차 1개만 필요한데 매 요청마다 50개 회차 전체를 역직렬화해야 한다는 것이었다.

```
[V1 흐름]
DB SELECT (PK 인덱스, 1ms) → 응답

[V2 흐름]
Redis GET (50개 묶음, 20만자) → JSON 역직렬화 (50배 비용) → 1개 추출 → 응답
```

- 회차 1개 응답하기 위해 50배의 역직렬화 비용 발생
- Redis → 애플리케이션 간 불필요한 대용량 데이터 전송
- 캐시 히트율 100%여도 역직렬화 오버헤드가 DB 조회 비용보다 훨씬 큼

<aside>


## ✅해결 방안

</aside>

벌크 캐싱에서 **단건 캐싱**으로 전환

```java
// 기존: 벌크 캐싱 (50개 묶음)
public List<EpisodeBulkCache> getBulkCache(Long novelId, int bulkIndex) {
    // 50개 회차 통째로 가져와서 역직렬화
}

// 변경: 단건 캐싱 (1개씩)
public EpisodeContentCache getContentCache(Long episodeId) {
    // 1개 회차만 가져와서 역직렬화
}
```

**캐시 키 변경:**

```
기존: episode_bulk::1::1  (50개 묶음)
변경: episode_content::1  (1개씩)
```

추가로 모든 회차를 캐싱하면 Redis 메모리가 무한정 늘어나는 문제가 있어서, **핫키 임계값(1시간 내 50회 이상 조회)을 넘긴 인기 소설 회차만 선별 캐싱**하여 메모리 효율도 확보했다.

</aside>

```
[회차 본문 조회 요청]
       ↓
단건 캐시 확인 (episode_content::{episodeId})
  └─ 캐시 히트 → 즉시 반환 (인기 회차)
       ↓
캐시 미스 → DB 조회 (PK 인덱스, 1ms)
       ↓
핫키 카운터 증가
  └─ 임계값(50회) 초과 → Redis에 단건 저장 (다음 요청부터 캐시 히트)
  └─ 임계값 미만 → 캐싱 없이 응답
```

<summary><b>전민우 - Mentorship.title → MentorFeedback 이동 — 도메인 설계 오류 발견 및 리팩토링</b></summary>
<aside>


## **💡배경**

</aside>

멘토링 도메인 초기 설계 시 `Mentorship` 엔티티에 `title` 필드를 추가했다. 멘토링 신청 시 제목을 입력받아 저장하는 구조였고, 당시에는 자연스러운 설계처럼 보였다.

코드 리뷰 과정에서 팀원이 `title`을 `MentorFeedback`으로 옮겨야 한다는 의견을 제시했다. 처음에는 `Mentorship.title`이 멘토링 신청 자체를 식별하는 제목이므로 현재 위치가 맞다고 판단했지만, 실제 도메인 구조와 피그마 화면을 다시 분석하면서 설계 오류를 발견했다.

---

<aside>


## 🚨문제 상황

</aside>

java

```jsx
// 기존 Mentorship 엔티티 — title이 신청 단위에 묶여있음
public static Mentorship create(Long mentorId, Long menteeId, Long currentNovelId,
                                String motivation, String manuscriptUrl, String title) {
    Mentorship mentorship = new Mentorship();
    mentorship.title = title;       // 멘토링 신청 전체의 제목
    mentorship.totalSessions = 0;   // 회차는 숫자만 있고 내용이 없음
    // ...
}

// 기존 MentorFeedback 엔티티 — 피드백 내용만 있고 회차 식별 불가
public static MentorFeedback create(Long mentorshipId, Long authorId, String content) {
    return new MentorFeedback(mentorshipId, authorId, content);
    // 몇 회차인지, 이 회차의 제목이 무엇인지 알 수 없음
}
```

`totalSessions`는 숫자 카운트만 있고 각 회차의 내용이 `MentorFeedback`에 담겨있지 않아 이력 관리가 불완전한 구조였다. 멘토 입장에서 최대 5명의 멘티를 동시에 관리하는 1:1 구조에서, 각 멘토링의 회차별 피드백 제목이 없으면 이력을 구분할 방법이 없었다.

<aside>


## 💭가설

</aside>

1. `title`이 표현하는 것이 **"멘토링 신청 전체의 제목"** 인지, **"각 피드백 회차의 제목"** 인지를 명확히 정의하면 올바른 위치가 결정될 것이다.
2. 피그마 화면에서 실제로 어떤 데이터가 표시되는지 확인하면 설계 방향이 명확해질 것이다.
3. `Mentorship.title` 제거 후 상세 조회에서 제목이 필요한 경우 `Novel.title`로 대체할 수 있을 것이다.

**Step 1: 도메인 책임 분리 원칙으로 위치 재정의**

피그마 화면 분석 결과, 멘토링 상세 화면에서 표시되는 것은 소설 제목이었고 별도의 멘토링 신청 제목은 존재하지 않았다. 각 피드백 회차를 구분하는 제목이 필요한 것이었으므로 `title`은 `MentorFeedback`에 있어야 한다는 결론을 내렸다.

## ✅배운점

</aside>

처음에는 팀원의 의견에 반대했지만 도메인 구조를 다시 분석하면서 생각이 바뀌었다. **"이 데이터가 무엇을 식별하는가"** 라는 질문 하나로 올바른 위치가 결정되었다. `title`이 멘토링 신청 전체를 식별하는 것인지, 각 피드백 회차를 식별하는 것인지를 명확히 하니 답이 나왔다.

설계 오류는 초기에 발견할수록 수정 비용이 적다. 이번에는 서비스 초기여서 테스트 코드 4개 수정으로 끝났지만, 서비스가 커진 후에는 훨씬 많은 곳을 수정해야 했을 것이다. 코드 리뷰와 팀원 간 논의가 설계 오류를 조기에 발견하는 가장 효과적인 수단임을 다시 확인했다.

또한 엔티티 시그니처 변경은 연쇄적으로 서비스, DTO, 테스트 코드까지 영향을 미친다. 변경 범위를 먼저 파악하고 순서대로 수정하는 습관이 중요하다는 것을 깨달았다.

<summary><b>정인호 - 웹훅 미수신으로 인한 결제 정합성 영구 불일치</b></summary>

## 📌 배경

본 프로젝트는 포트원(PortOne) V2를 연동한 포인트 충전 결제 시스템을 운영 중이다.

결제 흐름은 아래와 같다.

1. 프론트에서 포트원 SDK를 통해 결제 시작 → 우리 DB에 `PENDING` 상태 기록
2. 결제 완료 후 프론트가 `/confirm` 호출 → 포트원 상태 검증 후 `COMPLETED` 처리 및 포인트 지급
3. 포트원이 웹훅을 비동기로 전송 → 결제 상태를 재검증하여 불일치 보정

이 구조에서 웹훅은 상태 보완 수단으로 설계되어 있었다.

---

## ❗ 문제 상황

다음과 같은 시나리오에서 웹훅이 아예 도달하지 않거나, 처리에 실패할 수 있다.

- **네트워크 장애**: 포트원 서버와 우리 서버 사이의 일시적 단절
- **서버 다운타임**: 웹훅 수신 시점에 우리 서버가 재배포 중이거나 일시 장애 상태
- **재시도 소진**: 포트원의 재시도 정책(수회)을 모두 소진한 뒤에도 수신 실패

이 경우 포트원에서는 결제가 `PAID`로 처리됐지만 우리 DB는 `PENDING` 또는 `FAILED` 상태로 남아

**영구적인 정합성 불일치**가 발생한다.

사용자는 실제로 결제에 성공했음에도 포인트를 받지 못하는 치명적 상황이 생길 수 있다.

---

## 💡 가설

### 가설 1 — 웹훅만으로는 정합성을 보장할 수 없다

웹훅은 포트원이 "전송을 시도"하는 이벤트다. 수신 보장을 전제로 설계하면 단일 실패 지점이 된다.

외부 의존성이 있는 이상, 웹훅이 유실되는 케이스를 별도로 방어해야 한다.

### 가설 2 — 주기적 폴링으로 유실된 이벤트를 사후 보정할 수 있다

포트원 API는 `paymentKey`로 현재 결제 상태를 직접 조회할 수 있다.

일정 시간이 지나도록 상태가 갱신되지 않은 건들을 주기적으로 포트원에 직접 조회하면

웹훅 유실 여부와 무관하게 최종 상태로 보정할 수 있다.

---

## 🛠 해결 과정

### 1단계 — 보정 대상 기준 설계

단순히 "오래된 PENDING을 전부 조회"하면 정상 진행 중인 결제까지 건드릴 수 있다.

아래 두 가지 케이스를 보정 대상으로 정의했다.

| 대상 | 조건 | 사유 |
| --- | --- | --- |
| `PENDING` 30분 초과 | `createdAt < now - 30m` | `/confirm` 미호출 + 웹훅 유실이 동시에 발생한 케이스 |
| `FAILED` 30분~2시간 이내 | `createdAt between now-2h and now-30m` | confirm 타임아웃으로 잘못 FAILED 처리 후 웹훅도 유실된 케이스 |

> **30분 하한선을 둔 이유**: 방금 실패한 건은 정상적인 FAILED일 가능성이 높다.
> 
> 
> 30분이 지난 FAILED는 그 사이 웹훅이 도달했다면 이미 처리됐을 것이므로,
> 
> 미처리 상태로 남아있다면 웹훅 유실로 판단할 수 있다.
> 

### 2단계 — PaymentReconciliationService 구현

```java
@Scheduled(fixedDelay = 10, timeUnit = TimeUnit.MINUTES)
public void reconcile() {
    LocalDateTime now = LocalDateTime.now();

    // PENDING이 30분 넘도록 그대로인 건 — /confirm 미호출 + 웹훅 유실
    List<Payment> stalePending = paymentRepository
            .findByStatusAndCreatedAtBefore(PaymentStatus.PENDING, now.minusMinutes(30));

    // FAILED 30분~2시간 이내인 건 — confirm 타임아웃으로 잘못 FAILED 처리된 가능성
    List<Payment> recentFailed = paymentRepository
            .findByStatusAndCreatedAtBetween(PaymentStatus.FAILED, now.minusHours(2), now.minusMinutes(30));

    stalePending.forEach(p -> reconcileOne(p, "PENDING"));
    recentFailed.forEach(p -> reconcileOne(p, "FAILED"));
}
```

### 3단계 — 포트원 직접 조회 후 상태 보정

각 건에 대해 포트원 API를 직접 조회하여 실제 결제 상태를 기준으로 보정한다.

```java
private void reconcileOne(Payment payment, String reason) {
    io.portone.sdk.server.payment.Payment portOnePayment =
        paymentClient.getPayment(payment.getPaymentKey()).get(10, TimeUnit.SECONDS);

    if (portOnePayment instanceof PaidPayment paidPayment) {
        // 포트원은 성공, 우리는 PENDING/FAILED → COMPLETED + 포인트 충전
        paymentTransactionService.completePayment(...);

    } else if (portOnePayment instanceof FailedPayment || portOnePayment instanceof CancelledPayment) {
        // 포트원도 실패/취소 → FAILED로 정리
        paymentTransactionService.failPayment(payment.getId());

    } else {
        // 포트원도 아직 PENDING (진행 중) → 스킵
    }
}
```

### 최종 방어 구조

```
결제 실패/타임아웃 발생
│
├─ 1단계: cancelPayment() 내 compensateDeduct() ─── 즉시 복구 (포트원 상태 불명 시 보상)
│
├─ 2단계: 포트원 웹훅 도착 ────────────────────────── 수초~수십 초 내 보정
│         ├─ CancelledPayment + COMPLETED → REFUNDED + 포인트 처리
│         └─ PaidPayment + FAILED → COMPLETED + 포인트 충전
│
└─ 3단계: 재검증 배치 (10분 주기) ────────────────── 웹훅 유실 시 최후 방어선
          └─ 포트원 직접 조회 → 상태 보정
```

웹훅은 **"보완 수단"**, 재검증 배치가 **"보장 수단"** 이 된다.

---

## ✅ 결과

- `PENDING` / `FAILED` 상태로 방치되던 결제 건을 10분 이내에 자동으로 정합성 복구
- 웹훅 유실, 네트워크 장애, 서버 다운타임 등 모든 외부 요인에 무관하게 최종 일관성 보장
- 사용자가 결제 성공 후 포인트를 받지 못하는 케이스 제거

## 📚 배운 점

### 1. 외부 이벤트(웹훅)는 "최선 노력(best-effort)" 전달이다

외부 시스템이 전송하는 이벤트는 항상 유실 가능성이 있다.

이를 유일한 상태 동기화 수단으로 삼으면 단일 실패 지점이 된다.
웹훅은 빠른 반응을 위한 보완 수단이고, 주기적 폴링이 정합성을 **보장**하는 수단이어야 한다.

### 2. 재시도와 멱등성은 세트다

배치에서 동일한 결제 건을 여러 번 처리할 수 있으므로 `completePayment`, `failPayment` 등

모든 상태 전환 로직은 **멱등하게** 설계되어야 한다.
이미 `COMPLETED`인 건에 다시 포인트를 지급하지 않도록 상태 검사를 선행해야 한다.

### 3. 보정 대상의 시간 범위 설계가 핵심이다

"언제부터 언제까지"를 잘못 설정하면 정상 건을 건드리거나 실패 건을 놓칠 수 있다.

시스템의 응답 시간, 포트원 재시도 주기, 정상 결제 소요 시간을 함께 고려해서

하한/상한 기준을 신중하게 정해야 한다.

### 4. 결제 시스템은 3단계 방어를 기본으로 설계하라

즉시 보상(compensate) → 비동기 이벤트(webhook) → 주기 배치(reconciliation)

이 세 레이어가 갖춰져야 외부 의존성이 있는 결제 시스템에서 데이터 정합성을 신뢰할 수 있다.
